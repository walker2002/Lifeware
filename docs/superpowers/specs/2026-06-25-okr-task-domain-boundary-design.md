## 决策摘要 (TL;DR)

- **方向（宪法正确 + 代码史佐证）**：保持 OKR 与 Task 为两个独立 Domain；OKR 拥有 junction/贡献表；Task Domain 不感知 OKR。junction 方向被废弃代码的 TODO（「需要通过 junction 表重新关联」）直接佐证。
- **首版的错**：把「事件接线」当成近乎零成本的加法。实际跨域事件分发器**运行时不存在**（block F 仅声明性），所以事件驱动路径有 Nexus 级前置依赖。
- **历史关键事实**：`tasks.keyResultId` 曾存在、已被移除（遗留废弃处理器 + junction TODO）；而 `habits.key_result_id` 直接外键**仍保留**。这一不对称是本设计的核心待解问题。

## 问题陈述 (Problem Statement)

用户在重组 OKR Domain 时提出：OKR（尤其 KeyResult）与 Task 紧密关联（KR 进度由 task 完成推进），是否应合并为单一 Domain？若分开，耦合如何解开？

## 关键待解问题（需用户历史知识）

**为什么 `tasks.keyResultId` 被移除，而 `habits.key_result_id` 保留？** 这个历史决策是本设计最强的证据。可能性：
- (a) tasks 数量多/瞬态，直接外键造成紧耦合 → 故转向 junction；
- (b) 仅是未完成工作，habits 还没轮到迁移；
- (c) habits 是稳定 1:1，直连可接受而 tasks 不是。

用户大概率知道原因（项目近期提交）。它直接决定：habits 是否也要迁移到 junction（一致性），以及是否值得为事件驱动建跨域分发器。

## 决策方向（Direction — 宪法正确，代码史佐证）

保持分离；OKR 拥有 junction/贡献表；Task 不感知 OKR。理由：
1. **本体论不同**：Task 是执行单元（动词：能量/调度/生命周期）；OKR 是策略+度量层（名词：Objective=方向，KR=带 target/current/unit 的指标）。合并=把「被测量的东西」和「测量本身」焊死。
2. **KR 多态度量汇**：可被 task/habit/读书/体重推进；junction 的 `contributorType` 让 KR 不被任一来源绑架。
3. **Domain 隔离（Principle VI / Prohibition 3）**：OKR 不读 Task 内部数据。
4. **代码史佐证**：废弃 TODO 自身指向 junction 表。

## 方案对比（Approaches — 重新评级，诚实含落地成本）

### Approach A: 分开 + OKR 拥有 junction + 事件驱动（用户初选）
真实成本：**需先建跨域事件分发器**（Nexus 变更，§Nexus 不可侵犯性须讨论，不可静默添加）。优点：支持存储 `currentValue` + 阈值提醒/反应式外推。
- 方向 Completeness：9/10
- **含落地成本**：6.5/10（dispatcher 前置重）

### Approach B: 分开 + 读时聚合（Task Context Provider）
无需 Nexus 变更：Task 暴露 Context Provider（domain 级注册，仿 habits 已有 provider）→ OKR 读时聚合。`currentValue` 计算不落库。
- Completeness：7/10。**无需 Nexus 变更的最便宜首步**，但不支持阈值提醒。

### Approach C: 合并为单一 Domain
逆纹理、损多态性、唯一难逆。Completeness：4/10。

### Approach D: 暂不决定
但历史遗留（废弃处理器 + habits 不一致）须清理，不能完全不定。Completeness：6/10。

## 推荐落地路径（Corrected Sequencing — 关键修正）

鉴于跨域分发器不存在，**不应把 A 当成首步**。junction 数据模型先建，recompute 机制分阶段演进：

- **第 0a 步（清理，立即可做）**：删除 `app/actions/okr.ts` 废弃的 `handleTaskCompletedKRProgress`/`handleTaskDeletedKRRecalc`（错层 + 死代码 + 硬编码 MVP_USER_ID）；在 `manifest.md` 记录 `tasks.keyResultId` 移除历史。
- **第 0b 步（Tier-2 文档先行，宪法 §IV）**：`docs/usom-design.md`（Contribution 对象）+ `docs/database-design.md`（`contributions` 表）→ 再 `schema.ts`。代码先于文档是违宪。
- **第 1 步（首个真实 task→KR 需求，无需 Nexus 变更）**：建 junction/贡献表（OKR 私有）+ **读时重算**（Task 加 Context Provider，仿 habits）。`currentValue` 读时计算、不落库。避开 dispatcher。
- **第 2 步（当需要阈值提醒/反应式外推时）**：建跨域事件分发器（Nexus 变更，走宪法讨论）→ 演进到事件驱动存储 `currentValue`（完整 A）。
- **贯穿**：解决 habits/tasks 不一致（迁移 habits 到 junction，或文档化不对称理由，见 Open Question #1/#3）。

> 此排序保住用户选 A 的**方向**（OKR 拥有 junction，Task 无感），同时诚实面对 dispatcher 前置依赖：先读时聚合（B 机制，零 Nexus 变更），后事件驱动（A 机制）。

## 贡献模型（junction，OKR 私有）

```
contribution {
  id, userId,
  keyResultId,          // 指向 OKR 自己的 KR
  contributorType,      // 'task' | 'habit' | 'manual' | 'book' | ... （多态）
  contributorId,        // 来源 ID（如 taskId）—— 对 OKR 是不透明引用
  delta?,               // 非任务数单位时，单条贡献的增量（链接时设定）
  weight?,              // 可选权重
  createdAt, updatedAt
}
```

### 默认贡献规则（concrete）
- KR `unit === '任务数'`：`currentValue = count(linked tasks where status='completed')`。
- 其他单位（小时/本/斤）：`currentValue = sum(delta where linked source completed)`，`delta` 链接时由用户设定、默认 `targetValue / totalLinked`。
- **全量重推导（非增量）**：`currentValue` 永远从 `contributions + 来源当前状态` 重新计算 → 天然幂等。

### 并发与一致性
- **幂等**：全量重推导，事件/读时重算重放不重复递增。
- **并发**：`KeyResultRepository.updateProgress` 用乐观锁或事务内 `SELECT ... FOR UPDATE`，避免 lost update。
- **多租户（T-02）**：贡献查找与重算**必须**由事件/查询的 `userId` 限定；**禁止**沿用废弃代码的硬编码 `MVP_USER_ID`。
- **删除**：`TaskDeleted`/`TaskArchived` 触发回退；硬删场景由全量重推导兜底（贡献查不到来源 → 计 0）。

## 合规性（宪法对照）

- **Principle VI / Prohibition 3**：OKR 不读 Task 内部数据，只经 Context Provider（读时）或事件 payload（驱动）。✅
- **§III 业务事实写入口**：KR `currentValue`（FactField）落库必经写入口（Field Executor）。✅
- **§Nexus 不可侵犯性**：跨域事件分发器属 Nexus 变更，须经显式讨论，**不可静默添加**。⚠️ 这是 A 完整实现的宪法关卡。
- **§IV USOM 主权 + Tier-2 同步**：Contribution USOM 对象与 `contributions` 表须先入 `docs/`，再写代码。

## 成功判据 (Success Criteria)

- Task Domain 源码对 OKR 概念（keyResult/objective/okr）**零引用**（`grep` 验证，仅允许 compliance 测试）。
- KR 能被**至少一种非 task 来源**推进，且不需修改 Task Domain。
- `contributions` 表由 OKR Domain 独占；KR↔task 永远无 DB 外键（habits 处置见 Open Question #3）。
- 废弃 `handle*KRProgress` 已清理；无硬编码 `MVP_USER_ID` 残留。
- 跨域协作只经事件订阅 / Context Provider / Intent 三通道，无直连 Repository。
