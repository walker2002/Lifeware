## TL;DR

Phase 1 把 OKR Domain 的「数据模型 + 写入口」扶正到 [018]/[019] 范式，为 Phase 2/3 扫清违宪障碍，三件事：

1. **Cycle 升格为一级对象**：新建 `cycles` 表，`objectives.cycle_id` 为权威归属；**移除** objectives 的 `period_type/period_start/period_end` 列（DB 单一事实源），USOM `Objective.period` 改为**派生字段**（repo 读时 join cycle 填充）以零改读组件。
2. **扶正 §III 业务事实写入口**：新建 `createOkrsMutationService` 工厂 + manifest `mutation_mode` 三分类 + FactField 写走 FieldExecutor + 删 Repository 直写违宪点。区块 K CNUI Surface defer 到 Phase 3。
3. **OKR 范围 action/导航清理**：OKR 只暴露 `/okrs` 页面入口；lifecycle actions 收进工作台，不再作 GrowthMenu 菜单项。**不碰**他域导航。

- **period 的 UI 消费者**：写路径 `okr-form.tsx`(:24-209，用户设周期)/`okr-list`(:74)/`okr-workspace`(:59,82)；读路径 `okr-panel`(:92)/`okr-detail`(:140,162)/`okr-directory`(:58)/`hooks.ts`(:88,261)。
- **GrowthMenu**（`components/layout/growth-menu.tsx`）：渲染 tasks/timebox/habits/okrs 四域操作菜单，跨域主导航。

## 1. Cycle 对象升格（数据模型）

### 1.1 新建 `cycles` 表（OKR 私有，一级对象）

```
cycles:
  id            uuid PK
  user_id       uuid NOT NULL → users(id) ON DELETE CASCADE   -- T-01/T-02
  schema_version int NOT NULL DEFAULT 1
  cycle_type    text NOT NULL  -- enum: annual | quarterly | monthly | custom（对齐 022；迁移映射见 §4）
  name          text NOT NULL   -- 如 "2026-Q2"（可由 type+start 自动生成）
  period_start  date NOT NULL
  period_end    date NOT NULL
  status        text NOT NULL  -- enum: draft | not_started | in_progress | ended | reviewed（草稿/未开始/进行中/已结束/已复盘）
  created_at / updated_at       -- timestamptz
  -- 各状态时间戳：started_at / ended_at / reviewed_at（按生命周期补）
约束：
  check_cycles_period_end_after_start: period_end > period_start
索引：
  idx_cycles_user_status(user_id, status)
  idx_cycles_period(user_id, period_start, period_end)
```
> **健康度不落库**：`总体健康度` 读时聚合（§1.4），无 `health_score` 列。

### 1.2 objectives 表变更

- **新增** `cycle_id uuid NOT NULL → cycles(id) ON DELETE RESTRICT`（+ 索引 `idx_objectives_cycle`）。
- **移除** `period_type` / `period_start` / `period_end` 列、`check_objectives_period_end_after_start` 约束、`idx_objectives_period` 索引（约束/索引上移到 cycles）。

### 1.3 USOM 变更

- 新增 `Cycle` 类型（对齐上表字段；`period: { type; start; end }` 为 Cycle 自身字段）。
- `Objective`：新增 `cycleId`（权威）；`period` 改为**派生只读字段**（类型不变 `{ type; start; end }`），由 repo 读时 join cycle 填充——读组件零改动。

### 1.4 健康度读时聚合

```
cycle.health = aggregate( cycle 下所有 KR 的 progressRate )
默认聚合：均值（plan 可定加权/中位数）；Phase 2 引入 junction 后 currentValue 同样读时重算。
```

### 1.5 仓储变更

- 新增 `CycleRepository`（CRUD + findByUserAndStatus + findByPeriod）。
- `ObjectiveRepository.buildNumberPrefix` 改入参为 cycle（`cycleType + periodStart`）。
- `findByPeriod`/`findByStatusInPeriod` 改为 join cycles（按 `cycles.period_start` 过滤，经 cycle_id 关联）。
- `createOkrsGenericRepo` 扩展 `cycle` objectType 映射。
- mappers：`objectiveRowToUSOM` 读时 join cycle 填充派生 `period`；`objectiveUSOMToRow` 不再写 period 列。

### 1.6 Tier-2 文档先行（宪法 §IV）

**必须先改文档再改代码**：`docs/usom-design.md`（Cycle 类型 + Objective.cycleId + 派生 period 说明）+ `docs/database-design.md`（cycles 表 + objectives 变更）→ 再 `schema.ts` + 迁移。同步更新 `manifest.md` 版本历史。

## 2. 写入口扶正（§III 业务事实写入口合规）

### 2.1 新建 createOkrsMutationService

仿 `createHabitsMutationService`（~30 行）：
```ts
export function createOkrsMutationService(): DomainMutationService {
  const repos = createOkrsGenericRepo({
    objectiveRepo: new ObjectiveRepository() as any,
    keyResultRepo: new KeyResultRepository() as any,
    cycleRepo: new CycleRepository() as any,          // 新增
  })
  return createDomainMutationServiceFactory({
    domainId: 'okrs',
    repos,
    fieldUpdatedEventType: 'OkrFieldUpdated',          // 新事件类型
    repoLabel: 'OKRs',
  })
}
```

### 2.2 manifest field_metadata 补 mutation_mode 三分类

为 `objective` / `keyresult` / `cycle` 三对象的每个字段标注 `mutation_mode`（FactField / ContentField / PresentationField），对齐 domain-development-guide §2.1 跨字段红线。

### 2.3 删违宪直写

`app/actions/okr.ts` 中 `repo.save` / `repo.updateProgress` 等 FactField 直写改经 `mutation-service.update()`（单字段）/ `execute()`（聚合）。orchestrator 保留**仅**生命周期状态转换职责。

### 2.4 事件类型

`SystemEventType` 新增 `OkrFieldUpdated`（对齐 `TaskFieldUpdated`/`HabitFieldUpdated`）。

### 2.5 CNUI Surface（区块 K）defer 到 Phase 3

OKR 工作台 UI 复杂且 Phase 3 要重写，区块 K Surface 注册随重写落地，避免返工。

## 3. OKR 范围 action / 导航清理

- OKR 只暴露 **`/okrs` 页面入口**；lifecycle actions（activate/pause/complete/discard/archive/createObjective 等）收进工作台 UI 内触发，**不再作为 GrowthMenu 菜单项**。
- `/okrs/:id` detail route 折叠进 `/okrs` 工作台（plan 阶段定保留参数化还是全并）。
- **不碰** tasks/habits/timebox 三域在 GrowthMenu 的暴露——纯 OKR 范围。
- manifest `intent_triggers` / `view_routes` 相应收敛（plan 细化）。

---

## 4. 数据迁移

- 从现有 `objectives.period_type + period_start + period_end` 抽取 Cycle：按 `(user_id, type, start, end)` 分组 → 每组建一条 cycle 记录 → 回填 `objectives.cycle_id`。
- **cycle_type 映射**：annual/quarterly/monthly 直映；`semi_annual`/`daily`/`weekly` → `custom`（name 保留原值，避免语义丢失）。
- 回填完成后 DROP objectives 的 period 列 + 相关约束/索引。
- **幂等**：迁移脚本可重复执行（已建 cycle 跳过，按客观键去重）。
- 单测覆盖迁移正确性 + 幂等性。

- **Cycle**：CRUD + 状态转换 + health 读时聚合单测。
