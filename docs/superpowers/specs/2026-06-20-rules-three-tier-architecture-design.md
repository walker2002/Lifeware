## 1. 问题陈述（Problem Statement）

原始需求 `mydocs/dev/018-清晰SM管理范围的处理.md` 的「问题2」指出：当前 Domain 校验规则**散乱地存在于三处且彼此脱节**：

1. 交互层（CNUI/页面表单）：硬编码 inline 校验（如时长>0、必填、时间格式），且**部分规则只存在于交互中，可被绕过**。
2. 业务层（`onValidate`）：校验逻辑硬编码在 `hooks.ts` / `validation.ts`，且**只产出 `Passed`/`Rejected` 两个变体**，未利用已就绪的 5 变体判定模型。
3. 全局层（`RuleEngine`）：规则硬编码注册，非声明式驱动。

核心诉求：把规则**声明式化**，统一成 manifest 的 `rules:` 区块，让「交互层实时校验（L1）」和「业务层提交校验（L2）」从同一份规则源读取，消除「规则只在一层存在」的绕过风险与三处重复维护。

**不在本组范围**：L3 Nexus 全局规则（时间冲突/能量超载/目标过载/当日承诺上限/专注时间保护）—— 文档既定「不在本次修改范围」，保持现有 `RuleEngine` 不动（见 §7 开放问题 Q1 的边界澄清）。

**关键 reframe**：本组要新建的东西不多 —— **底层管线全就绪了**（5 变体、聚合排序、suspend 路由、onValidate→Orchestrator、RuleEngine→ValidationResult 映射）。真正缺的只有两样：

1. **声明式规则源**：manifest `rules:` 区块 + USOM/loader 类型 + 一个「规则 ID → 检查函数」的 registry 框架。
2. **两个消费者改接线**：`onValidate` 从硬编码改成调统一规则评估；CNUI/表单从 inline 改成读 manifest rules 跑 realtime。

这让全部风险收敛到**一个点**：规则模型怎么设计 —— 一条 manifest 规则如何同时被**同步的客户端 CNUI**（realtime，不能查库）和**异步的服务端 onValidate**（submit，可查 repo）消费。

## 3. 前提（Premises，已与用户确认）

- **P1**：L3 全局规则不在本组（文档既定），现有 `RuleEngine` 行为不变（§7 Q1 的边界例外除外）。
- **P2（机制选型）**：走「**声明式绑定 + 命令式处理器**」—— manifest 只声明规则元数据（id/phase/fields/message），检查函数仍是 TS 代码（复用现有 `RuleEngine` Rule 对象模式）。**不走全 DSL**（表达式解释器是过度设计，且「关联目标存在」这类查库规则 DSL 表达不了）。
- **P3（同步/异步边界）**：`phase: both` 规则的 realtime 部分**必须纯函数、同步、不查库、不读 now**（客户端跑）；只有 `phase: submit` 规则才允许 async 查 repo。`phase` 由此具备强语义约束（both=同步纯函数可客户端跑；submit=可异步查库），由 registry 的函数签名在类型层强制。
- **P4（范围边界，用户 2026-06-20 裁定 = C1，评审 P4 精确化）**：规则层**四域全覆盖**，但**不触碰 okrs/timebox 的写入口编排债** —— 即不做「裸 `repo.save`→`domainMutationService` 编排」重构（那是 Group 1 既定推迟的独立切片 = 用户否决的 C2）。**精确边界**：onValidate 内的**只读 repo 查询**（如 okrs `activateObjective` 的 KR 数量前置检查）**可随规则迁入 registry**（作为 `phase: submit` 规则，经 `ServerRuleCtx.repos` 查库）；**不动的是写入口编排**（字段写/状态写仍走现状）。timebox 的 CNUI submit 仍是 stub，realtime 规则现在可用、submit 规则定义好等 submit 落地才触发。

## 4. 规则模型设计（核心）

### 4.1 manifest 声明（声明式绑定）

在四域 `manifest.yaml` 顶层新增 `rules:` 区块，每条规则声明元数据，**不含逻辑**：

```yaml
rules:
  - id: habit_title_required
    phase: both               # 客户端 realtime 提示 + 服务端权威（同一套 check 经适配器重跑）
    fields: [title]           # both ⟹ 单字段（不变式见 §4.2）
    message: "习惯名称不能为空"

  - id: habit_frequency_type_valid
    phase: both               # 单字段枚举合法（frequencyType ∈ daily/weekly/custom）
    fields: [frequencyType]
    message: "频率类型必须是 daily/weekly/custom"

  - id: habit_frequency_days_required
    phase: submit             # 多字段规则只能 submit：weekly 时 daysOfWeek 必填
    fields: [frequencyType, daysOfWeek]
    message: "weekly 频率必须指定星期"

  - id: habit_linked_goal_exists
    phase: submit             # 仅 submit（需查库验证目标存在）
    fields: [keyResultId]
    message: "关联的目标/关键结果不存在"
```

字段语义：

| 字段 | 取值 | 含义 |
|---|---|---|
| `id` | 全域唯一字符串 | 规则标识，绑定 registry 中的检查函数（id 完整性校验见 §4.2） |
| `phase` | `both` \| `submit` | 触发时机：`both`=客户端 realtime 提示 + 服务端权威；`submit`=仅服务端权威（需查库或多字段） |
| `fields` | 字段名数组 | 该规则关注哪些字段；`both` 必须单字段（不变式见 §4.2） |
| `message` | string | 面向用户的提示文案（§7 Q4 i18n 留口） |

> **无 `level` 字段、无 `realtime`-only phase**（plan-eng-review 2026-06-20 修正，见 §4.2 与 §12 修订记录）。L1/L2/L3 是架构执行位置的分层（存活于宪法/usom-design），不是三套不同规则；manifest 只需声明「何时跑」（phase）+「能否查库」由 check 函数签名（同步/异步）天然区分。realtime 永远是权威层的**加法早提示**，不存在「只在交互层、可被 submit 绕过」的规则——这正是需求 problem 2 要消灭的病灶。

### 4.2 执行模型（phase 驱动，`level` 已弃用）

规则只有两种 phase，**二者都在权威层执行**（域规则→L2 onValidate；全局规则→L3 RuleEngine，聚合于 Orchestrator）；realtime 只是 `both` 规则的**附加**客户端早提示，从不替代权威：

| phase | 客户端 realtime 提示（L1，附加） | 服务端权威（L2 onValidate / L3 RuleEngine） |
|---|---|---|
| `both` | ✅（单字段 blur，跑同一套 RealtimeCheck） | ✅（经适配器重跑同一套 check） |
| `submit` | ❌（多字段或需查库，blur 时无法判定） | ✅ |

**核心不变式**（消灭绕过 + 同步/异步边界；由 `scripts/validate-manifest.ts` build/CI 校验 + 插件注册期断言）：

1. **无 realtime-only**：每条规则都在权威层执行；realtime 是 `both` 的附加提示，不替代权威。故 phase 无 `realtime` 取值——杜绝「只在交互层、可被绕过」的规则（需求 problem 2 的病灶）。
2. **`phase: both` ⟹ 单字段**：多字段规则（时间窗、依赖关系、跨字段约束）只能 `submit`——blur 单字段时其余字段未必就绪，无法判定。
3. **`phase: both` 的 RealtimeCheck 必须同步纯函数**：不查库、不读 `Date.now()`/随机，同输入同输出（类型层 + 测试约束，§4.3）。
4. **id 完整性**：每个 manifest `rule.id` 必须在对应域 registry 注册了 check；孤儿 id = `scripts/validate-manifest.ts` 报错（build/CI 阻断）+ 插件注册期断言。注：`loadDomainManifest()` 是纯 YAML loader、无法触及 TS registry，故校验落 build 脚本 + 插件注册期（非 loader 函数内）。

> 原 `level × validationPhase` 2×3 矩阵已弃用（plan-eng-review 2026-06-20）：`level` 冗余（`ui` 与 `domain+realtime` 在权威层毫无差别，都重跑或省略），且其 `ui` 取值制造了「可绕过规则」——违需求。同步/异步签名已足够区分 client/server 能力。

**端到端规则流转图**（canonical，实现时内联精简版于 `evaluateDomainRules` 源码）：

```
manifest.yaml ── rules: 声明 id / phase / fields / message ──────────┐
                                                                      │
 rules-registry.ts ── id → RealtimeCheck(sync) | SubmitCheck(async) ──┤
                                                                      │
   ┌──────────────────────────────────────────────────────────────────┤
   │                                                                  │
  CNUI realtime 提示 (L1, 附加)              onValidate submit (L2 权威)
  useManifestRules()                         evaluateDomainRules(intent, serverCtx)
  validateField(field, value) 单字段          跑 phase∈{both,submit} 规则
  ClientRuleCtx (最小, 无 now)               ServerRuleCtx {repos, userId, now}
   │  fail-OPEN (客户端吞错+记日志)            │  fail-CLOSED (SubmitCheck 抛错→Rejected)
   └── FieldIssue[] {field,message}           ├── both 规则: 适配器重跑 RealtimeCheck
                                              ├── submit 规则: SubmitCheck → ValidationResult(5变体)
                                              └── aggregateValidation(VALIDATION_RANK) 取最严格
                                                          │
                          Orchestrator: Passed → 写入口 / suspend(PWW/NeedInput/NeedConfirm) / end(Rejected)
```

### 4.3 registry（命令式处理器）

每个域新增 `rules-registry.ts`，按 id 注册检查函数。**用函数签名在类型层区分同步/异步**，强制 P3：

```ts
/** 字段级问题：realtime 检查产出，field 用于 §4.4 提交失败按字段回填标红。 */
interface FieldIssue {
  field: string
  message: string
}

/** realtime 检查：同步、纯函数、无 repo、不读 now/随机。按【单字段值】判定。 */
type RealtimeCheck = (value: unknown, ctx: ClientRuleCtx) => FieldIssue[]

/** submit 检查：可异步、可查 repo。按【整个 Intent】判定（多字段/查库规则进这里）。 */
type SubmitCheck = (intent: StructuredIntent, ctx: ServerRuleCtx) => Promise<ValidationResult>

/** 客户端 realtime 上下文：最小化，不带 now（C3：realtime 只硬错误，无需时序）。 */
interface ClientRuleCtx {
  /* 仅透传规则元数据；刻意不携带 now/userId，保证纯函数 */
}

/** 服务端 submit 上下文：userId 强制多租户过滤（T-01~T-04）；now 取自 snapshot.currentTime。 */
interface ServerRuleCtx {
  repos: DomainRepos
  userId: USOM_ID
  now: number
}

interface DomainRuleRegistry {
  realtime: Record<string, RealtimeCheck>  // key = rule id；仅 phase: both 规则注册
  submit: Record<string, SubmitCheck>      // key = rule id；phase: submit 规则 + both 规则的权威重跑
}
```

- realtime 检查产 `FieldIssue[]`（字段级、轻量、可直接喂表单 error state）。
- submit 检查产 `ValidationResult`（复用已就绪 5 变体），经 `aggregateValidation`（现有）按 `VALIDATION_RANK` 聚合，再与 L3 `RuleEngine` 结果在 Orchestrator 合流 —— **全程复用现有管线，零改动**。

现有散落的 `validateHabitFields` / `validateTaskFields` / okrs 内联校验**重构进 registry**，不是另写一套。

**realtime（单字段）与 submit（多字段/查库）的分工（plan-eng-review 2026-06-20 定稿）**：

- **`phase: both` 规则**（必单字段）只注册一套同步纯函数 `RealtimeCheck`：realtime 阶段产 `FieldIssue[]`；submit 权威阶段经适配器 `fieldIssuesToValidationResult(issues)`（空→Passed，非空→Rejected 带 errors）**重跑同一套 check**，不另写 submit 版本。
- **`phase: submit` 规则**（多字段或需查库）注册 `SubmitCheck`，按整个 Intent 判定，可返 5 变体任一。**多字段规则天然进 submit**——blur 单字段时其余字段未必就绪，无法判定（故 §4.2 不变式 2：`both ⟹ 单字段`）。
- **聚合语义**：适配器输出（Passed/Rejected）与 submit 规则的 5 变体结果**一起**喂 `aggregateValidation`，按现有 `VALIDATION_RANK` 偏序取最严格（如某 submit 规则产 `NeedConfirm`、某 both 规则适配产 `Rejected`，则 `Rejected` 胜出）——复用现有聚合，零新规则。
- **异常策略（不对称，Codex P1-7 修正）**：客户端 realtime 检查 **fail-OPEN**（抛错→吞+记日志+延后到 submit，不崩 onBlur handler）；服务端 submit 检查 **fail-CLOSED**（SubmitCheck 抛错→该条计 Rejected「规则校验失败，请重试」+ 记日志）。realtime 是附加提示，坏它不可阻断用户；submit 是权威，坏它宁可阻断也不放过无效数据。

### 4.4 两个消费者改接线

**消费者 A — onValidate（L2）**：从硬编码改成调统一入口
```ts
// domains/<域>/hooks.ts — onValidate
const result = await evaluateDomainRules(domainId, intent, serverCtx);
return result;  // 已聚合的 ValidationResult
```
`evaluateDomainRules` 读 manifest（取 phase ∈ {submit, both} 的规则）→ 查 registry 跑检查 → `aggregateValidation` 聚合。（`domainId` 由各域 `hooks.ts` 闭包绑定本域 id；client 侧 `useManifestRules` 的 `domainId` 由渲染表单的 Server Component props 透传。）

**消费者 B — CNUI/表单（L1）**：从 inline 改成读 manifest
```ts
// 通用 hook，所有域表单复用
const { errors, validateField } = useManifestRules(domainId, clientCtx);
// validateField(field, value) 同步跑命中该字段的 phase: both 规则（单字段）
```
- realtime 阶段：blur 时 `validateField` 同步跑 fields 命中该字段的所有 `phase: both` 规则，写 `errors` state。
- submit 前：客户端把 `phase: both` 的规则全跑一遍做**尽力而为的预检**（提升体验、减少提交失败），失败则阻止 submit；服务端 onValidate 始终权威兜底。
- **submit 失败回填（评审 P3）**：客户端 realtime 预检通过、但服务端 onValidate 返回 Rejected（如 submit 规则查库发现「关联目标不存在」）时，服务端 errors 按 `fields` 名映射回 `useManifestRules` 的 `errors` state（匹配到具体字段则标红该字段）；匹配不上的走表单级提示（toast/顶部 banner，遵循 `docs/UI-DESIGN-SPEC.md`）。

### 4.5 client/server 数据流：CNUI 如何拿到 rules（评审 P1，阻塞项）

`loadDomainManifest`（`domains/manifest-loader/loader.ts`）`import fs from 'node:fs'`，是 **server-only**；而 `habit-form.tsx` / `okr-form.tsx` / `timebox-draft-editor.tsx` 都是 `'use client'` 组件 —— 客户端**不能**直接 import loader。R0 必须落地其一：

- **方案 A（推荐）**：realtime 规则的**元数据**（id / fields / message）由渲染表单的 Server Component 用 `loadDomainManifest` 提取后，作为可序列化 props 传给 client 表单组件；realtime **check 函数**本身是 TS 代码，放在 client 可 import 的 registry 子集里（直接 import，不经网络）。即客户端拿到的是「**元数据 via props + check 函数 via import**」，不是整份 YAML。
- **方案 B**：每域加一个 server action `getRealtimeRules(domainId)` 返回元数据，client hook 调用。

两种方案都只把 **realtime 规则子集**（元数据）送到 client；submit 规则（可能查库）绝不进 client bundle。R0 选 A 起步（props 传递最简，无需新 server action）。

## 5. 方案对比（Approaches Considered）

### Approach A：纵切试点 —— habits 单域端到端
⑦类型+loader+registry 框架（轻量）→ ⑧只 habits 规则 → ⑨habit-form 读 rules → ⑩habit onValidate 调 rules。tasks/okrs/timebox 后续复制。Effort M，Risk Low。Completeness 6/10（机制完整、覆盖窄）。

### Approach B：结构先行 —— 完整类型系统 + tasks/habits 双域
⑦ 类型/loader/registry 做厚（四域预留）+ 两域规则全改接线。okrs/timebox 推迟。Effort L，Risk Med。Completeness 8/10。

### Approach C1（✅ 用户选定）：规则层四域覆盖，写入口债挡在范围外
四域 manifest 加 `rules:` + 类型 + loader + registry；四域 L1+L2 规则定义+实现；四域 onValidate 改调 `rules()`；四域 CNUI/表单改读 rules。okrs onValidate 只加规则不动写入口架构；timebox realtime 现在可用、submit 规则待落地。**不做 okrs 写入口重构**（独立切片）。Effort XL（大但边界清晰），Risk Med-High（四域同时改 onValidate+表单，回归与 CNUI 自检面大）。Completeness 8/10（四域规则完整、写入口债留作独立切片）。

### （已否决）Approach C2：C1 + okrs/timebox 写入口债
含 okrs 裸 `repo.save`→编排 重构 + timebox submit stub 落地 + 两域 mutation_mode 完整补齐。用户 2026-06-20 否决 —— 把 Group 1 既定推迟的 okrs 重大重构塞进本组会范围失控，okrs 写入口应走自己的独立设计。

## 6. 推荐执行方式（Recommended Approach）= C1，但分 5 个切片推进

C1 是 XL，**不能一次性提交**。沿用 Group 1 已验证的「试点→机制→复制」节奏，拆成 5 个独立可评审切片（每切片 subagent-driven-dev + TDD + 双评审 + Tier-2 文档同步）：

- **R0 机制地基（walking skeleton，plan-eng-review 2026-06-20 修正）**：USOM `rules` 类型 + `manifest-loader` schema（新增 `rules:` 字段）+ §4.2 不变式校验（落 `scripts/validate-manifest.ts` build/CI + 插件注册期断言，含 id 完整性）+ 通用 registry 框架（`RealtimeCheck` 单值 / `SubmitCheck` + `FieldIssue`/`ClientRuleCtx`/`ServerRuleCtx` 类型 + `evaluateDomainRules` + `useManifestRules` + `fieldIssuesToValidationResult` 适配器）+ 宪法 Tier-2 新增「规则三层」概念。**不再「无域规则」**——而是落 ≥1 条真实 `phase: both` 单字段 tracer 规则（realtime 经 `useManifestRules` 消费）+ ≥1 条真实 `phase: submit` tracer 规则（经 `evaluateDomainRules` 消费），用真实规则打通两层、锁定模型（而非合成 fixture）。tracer 落域见 §7 Q1/Q6（避免与 R1 golden 冻结冲突）。这是风险最高的切片（模型定型在此）。
- **R1 habits 端到端（试点）**：habits manifest `rules:` + 把 `habits/validation.ts` 重构进 registry + 改 habits onValidate + 改 `habit-form` 用 `useManifestRules`。**第一个真实规则，验证 R0 框架**。CUC-01~12 自测。
- **R2 tasks 端到端**：同 R1，对 tasks。重构 `tasks/validation.ts`。
- **R3 okrs 规则层**：okrs manifest `rules:` + 把 okrs 内联 onValidate 校验重构进 registry + 改 `okr-form`。**严守 P4：onValidate 保持 repo 形状，只加规则，不动裸 `repo.save`。**
- **R4 timebox 规则层**：timebox manifest `rules:` + 改 `timebox-draft-editor`（realtime）。submit 规则定义但 dormant（submit stub 未落地）。**§7 Q1**：决定 `rule-engine/rules/timebox.ts` 的字段级规则迁移边界（见 Q1）。
- **每切片**：usom-design §4.4 + manifest + 宪法同步（Tier-2 强制，见 memory `feedback-tier2-sync`）。

**迁移等价性护栏（评审 P5，每切片强制）**：迁移前把各域 **onValidate/hooks 输出**（errors 文案、顺序、边界值）冻结为 golden test，迁移后必须**逐字通过**同一组测试。**范围（plan-eng-review 2026-06-20 精确化）**：golden 只冻结 onValidate(L2 hooks) 输出；从 L3 **搬迁**过来的规则（如 timebox DurationRange 的 warning→Rejected）属「纠正规则归属」的**故意行为修正**，其旧测试须**更新**（非冻结）——否则会阻断修正。

每切片独立 PR，可回退；R0→R1 是关键路径；**R2-R4 须在 R1 sign-off 后放行**（非仅 R0）——模型风险集中在 R0/R1，R1 可能逼改契约，对着 R0 签名起的 R2-R4 工作会返工。

## 7. 开放问题（Open Questions）

- **Q1（L3 边界 + 迁移阶段，plan-eng-review 2026-06-20 逐条裁决）**：现有 `nexus/core/rule-engine/rules/timebox.ts` 导出 **4 条** Rule，按新模型（§4.2：`both ⟹ 单字段`，无 realtime-only）逐条定归属：
  - **FieldCompleteness**（title/startTime/duration 多字段）→ **迁，但分解**为 3 条单字段 `phase: both` 规则（`timebox_title_required` / `timebox_start_time_required` / `timebox_duration_required`），各可 realtime 提示。
  - **DurationRange**（duration 单字段）→ **迁为 `phase: both`**；其 warning→Rejected（"建议5分钟"→"必须5..480"）是「纠正归属」的故意行为修正，旧 `timebox-rules.test.ts` 须更新非冻结。
  - **StartTimeInFuture**（startTime 单字段）→ **迁为 `phase: submit` 返 PWW**（Q6 已定）：保「允许排过去+警告」语义、零行为变更。服务端 `snapshot.currentTime` 提供 now；**`ClientRuleCtx` 无需 now**（realtime 保持纯）。
  - **DelayedStartRule**（startTime + action=`start_timebox`，>30min 警告）→ **不迁，留 L3**：动作作用域的时序行为规则，非字段规则，塞不进 manifest `fields` 模型。
  - **对照锚点** `timebox-overlap.ts` TimeOverlap（startTime+duration 查库、跨 timebox 冲突）→ **不迁，纯 L3 全局**。
  - **净结论：3 迁（FieldCompleteness 分解、DurationRange、StartTimeInFuture）+ 2 留（DelayedStart、TimeOverlap）**。迁移以「纠正规则归属」名义做，须文档说明（非「改 L3 行为」）。
- **Q2（mutation_mode 补齐，评审 P6 收紧）**：R3/R4 触碰 okrs/timebox 的 manifest，是否顺带补齐其 `mutation_mode` 分类？**默认不补齐** —— mutation_mode 牵动字段执行器 FactField/ContentField 写路径分发，属写入口范畴，与本组无关；保持 P4 边界纯净。若要补，须先用独立 micro-切片验证「只声明 mutation_mode、不改执行器分发」零回归，且不得并入 R3/R4。
- **Q3（realtime UX）**：realtime 规则的 UI 表现（inline 字段错误 / 提交按钮置灰 / toast）须遵循 `docs/UI-DESIGN-SPEC.md`，且新增/改 CNUI Surface 必过 CUC-01~CUC-12 自测（memory `feedback-cnui-checkpoints`）。
- **Q4（文案 i18n）**：manifest `message` 内联中文，后续若上 i18n 需抽 key。本组先内联，留 `id` 作 i18n key 预留。
- **Q5（规则复用跨域）**：如「时长必须>0」tasks/timebox 都有。registry 是否抽公共 check（如 `makePositiveNumberCheck(field)` 工厂）？**plan-eng-review 2026-06-20：推迟到 R1/R2**——R0 即使有 2 条 tracer，2 条不构成 DRY 触发线（≥3 才抽），R0 建工厂仍是早抽象。等 R1/R2 真出现 tasks/timebox 重复时再抽。
- **Q6 ✅ 已决（submit/PWW，`ClientRuleCtx.now` 砍除）**：StartTimeInFuture 迁为 `phase: submit` 返 PWW——保「允许排过去+警告」语义、零行为变更。**`ClientRuleCtx.now` 砍除**（无 realtime 规则需要 now，realtime 保持纯函数零外部依赖）；服务端 `snapshot.currentTime` 提供 now。代价：放弃 timebox realtime 编辑面的 inline 过去时间提示（到 submit 才弹警告卡），换取 realtime 纯度 + 零行为变更。
- **Q7 ✅ 已决（专用 fixture 域）**：R0 tracer 落专用 fixture 域（真实 `manifest.yaml` + registry，仅框架验证用，不碰真实四域），打通 loader→registry→两消费者。R1-R4 再对真实四域迁移。R0/R1 干净分离、无 golden 冻结纠葛；fixture 域作为框架一致性测试永久保留（非抛弃）。

## 9. 风险（Risks）

- **规则模型定型风险（最高，集中在 R0/R1）**：`phase` 模型（both⟹单字段、无 realtime-only）与同步/异步签名若设计不当，R2-R4 复制时返工。缓解：R0 walking-skeleton 用真实 tracer 打通两层、R1 单域验证后再放量；R2-R4 须 R1 sign-off 后放行。
- **回归面大**：四域同时改 onValidate + 表单。缓解：分切片独立 PR，每切片 TDD + 双评审 + base/head 失败集合对比。
- **CNUI 自检遗漏**：表单改接线易踩三层嵌套/重复标题等老问题。缓解：每涉及 CNUI 的切片强制 CUC-01~CUC-12。
- **L3 边界越界（Q1）**：迁移 timebox.ts 规则可能被理解为「动了 L3」。缓解：Q1 交 `/plan-eng-review` 明确，迁移以「纠正规则归属」名义做并文档说明。
