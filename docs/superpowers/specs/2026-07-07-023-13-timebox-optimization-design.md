## Problem Statement

[023.12] lifecycle 简化（3 状态 + revert 闭环）已 ship，但遗留 5 项待办（见 `mydocs/dev/023.13-TimboxDomain相关优化.md`）：

1. **TD-019 预防闭环**：[023.12] ship 时 `STATUS_TRANSITION_ACTIONS` 白名单漏注册 `revertTimebox/revertAppointment`，导致 /timeboxes 任意 logged/cancelled 卡片点「回退」100% 阻断。bug 本身已 hot-fix（`0ead95c`），但**手工维护白名单的漂移根因未除**——下次新增 lifecycle transition 还会踩。
2. **回退 Bug 验证**：用户报告 `timebox.ts:197` 兜底「回退时间盒失败」。错误签名与 TD-019 根因吻合，**疑似已随 hot-fix 修复**，需验证。
3. **/timeboxes 打卡体验**：现状 planned 卡只有「打卡（simple log）/取消/删除」，缺批量打卡、一键打卡；logged/cancelled 缺「一键回退」；「打卡」按钮应升级为开抽屉做**详细打卡**（实际时间窗 + 深度专注时长 + 实际能量消耗 + 执行详情）。
4. **/logTimebox CNUI**：现仅三态(完成/未完成/跳过)+备注，需与 /timeboxes 打卡专区**共享字段**。
5. **月历翻页**：DayView 右栏 MiniCalendar 只显当月，无上下月翻动。

## What Makes This Cool

打卡不再是「点一下变 ✓」的薄记号，而变成**一次有数据密度的执行度量**——实际 vs 计划时间窗、深度专注时长、本次能量消耗、执行详情。这是「成长陪跑教练」能做行为建模的粉底：每个时间盒都留下一份可复盘的执行样本。同时 TD-019 预防闭环（白名单自动派生 + pre-push 校验）把一个反复踩的 process-level 坑永久填掉。

## Constraints

- **[023.12] 决议保留**：3 状态 lifecycle（planned/logged/cancelled），running/overtime/ended 读时派生不持久化。
- **D8 能量维度决议**：`EnergyCost` 4 维只活在 archetype 侧，业务表严禁存 4 维模型（memory: 能量维度宪章缺口）。
- **JSONB schema 演进**：`timeboxes.execution_record` 是 JSONB，加字段免 DDL 列变更，但 mapper 必须向后兼容旧记录（新字段 undefined）。
- **手写 SQL 迁移**：项目 drizzle snapshot 坏了（memory），迁移一律手写 + psql + 登 journal。本设计的字段演进若纯 JSONB 形状变更则**无需新迁移**。
- **CNUI 双注册**：新增/改 surface 须 server `surfaceHandlers` + client `register-client-surfaces` + manifest K-block 四路闭合（memory）。

## Premises（本 session 拍板）

1. **切割方式**：5 项**一个大 PR 全做**（用户明选）。§3/§4 数据模型决策本轮一并落。
2. **P2 数据模型**：扩展 `DetailedExecutionRecord` + 单值 `energyActual`（非 4 维）。加 `actualStartTime/actualEndTime/focusMinutes/energyActual`；任务执行详情复用 `notes`。能量用「单次度量标量」绕开 D8 4 维禁令，**不动宪章**（业务表存的不是 EnergyCost 模型，是一次 reading）。
3. **P3 回退守卫**：logged 卡「回退」走 **AlertDialog 披露数据丢失** → 确认后 clear executionRecord + SM revert。调和 [023.12] AM7（不"悄悄"丢，明示丢）。cancelled 无 executionRecord 仍真·一键。

## Approaches Considered

### Approach A: 扩展记录 + 单值能量 + 确认弹窗回退（**采纳**）
- USOM: `DetailedExecutionRecord` += 4 可选字段；DB JSONB 形状演进免迁移。
- 能量默认取 archetype 4 维均值（`getDefaultEnergyActual` helper）。
- 回退：前端 AlertDialog → confirmed 后 clearExecutionRecord + revert 顺序调用。
- §1: A1 白名单从 manifest 派生 + A2 pre-push validator；§5: MiniCalendar 内部 viewMonth state。
- **Effort: M（~3-4 天 CC）**；**Risk: Low-Med**（JSONB 兼容 + 回退守卫语义变更需测）。

### Approach B: 完整 4 维能量 + 静默回退
- 能量存 4 维 EnergyCost（破 D8，需宪章 amendment）；回退静默清记录（推翻 [023.12] AM7）。
- **Effort: L**；**Risk: High**（治理成本 + 逆转上轮产品决策）。本轮过重，未采纳。

### Approach C: 最小映射，新字段 defer
- 只落 actualDuration + notes；深度专注/能量 defer 成 TD。
- **Effort: S**；但用户明确要的字段丢了，打卡专区瘦成「时间窗+备注」。未采纳。

## Recommended Approach

**Approach A**——满足全部 5 项需求，不破 D8/AM7 既有决议，JSONB 演进零迁移风险。

## 详细设计

### §1 TD-019 预防闭环（A1/A2 + L2）

**A1 白名单自动派生**（`domains/timebox/rules-registry.ts`）：

⚠️ **命名空间陷阱（reviewer 抓到的 blocker，plan 期修正）**：`STATUS_TRANSITION_ACTIONS` 的成员是 **intent-namespace**（`logTimebox`/`revertTimebox`/`cancelAppointment`…）。plan 期读 `buildActionMap`（`nexus/orchestrator/lifecycle-configs.ts:59-109`）发现：intent-namespace action **本就由 lifecycle transitions 派生**——对每条 `lifecycle[objectType].transitions[*].action` 生成 `${action}${PascalCase(objectType)}`（如 `log`+`Timebox`→`logTimebox`，`revert`+`Appointment`→`revertAppointment`）。这才是正确派生源，**无需改 manifest 加标记**。

- 新增纯函数 `buildStatusTransitionActions(): Set<string>`：复刻 buildActionMap 的 lifecycle 派生逻辑，但 (a) **排除 `create` action**（create 需字段必含校验，不跳过），(b) 只取 intent-namespace camelCase 形态（不要 snake_case legacy 形态）。逻辑：
  ```
  for each (objectType, def) in timebox-manifest.lifecycle:  // timebox + appointment 两 objectType 同 manifest
    pascal = PascalCase(objectType)  // timebox→Timebox, appointment→Appointment
    for t in def.transitions:
      if t.action === 'create': continue
      set.add(`${t.action}${pascal}`)
  ```
  产出（[023.12] 后）：`{logTimebox, cancelTimebox, revertTimebox, cancelAppointment, completeAppointment, revertAppointment}`。
- `STATUS_TRANSITION_ACTIONS` 由手工常量 Set 改为模块加载时 `buildStatusTransitionActions()` 派生产物。
- **死成员自动清理**：现 Set 里的 `startTimebox`/`endTimebox`/`overtimeTimebox`/`startAppointment`/`expireAppointment` 在 [023.12] lifecycle 简化后已无对应 transition——派生后自动消失（plan 期需 grep 确认这些 action 名已无 dispatch 调用方，否则补 transition）。
- **appointment 域同机制收敛（issue 7）**：`appointmentFieldsValid`（rules-registry.ts:239）现硬编码 `!== 'createAppointment' && !== 'editAppointment'` skip，分叉易漂移。改为 `STATUS_TRANSITION_ACTIONS.has(intent.action)`（buildStatusTransitionActions 已含 appointment 的 cancel/complete/revert）——两域同 Set 同机制，drift 单一源头。注：`editAppointment` 不是 lifecycle transition（无对应 transition，走字段校验），现状 `!== 'editAppointment'` 的语义由"editAppointment 不在派生 Set 里"自动满足。
- **避免循环依赖**：`buildStatusTransitionActions` 不 import orchestrator（会环），改用 `loadDomainManifest('timebox')`（manifest-loader，叶子模块）直接读 lifecycle。

**A2 pre-push validator**（`validate:rules-registry`）—— 显式接线（issue 2）：
- 新建 `frontend/scripts/validate-rules-registry.ts`：独立实现同一派生逻辑（读 timebox manifest lifecycle，排除 create，派生 intent-namespace Set），对比 `rules-registry.ts` export 的 `STATUS_TRANSITION_ACTIONS`，drift → 退出非零 + 打印缺失/多余成员。
- `frontend/package.json` 加 `"validate:rules-registry": "npx tsx scripts/validate-rules-registry.ts"`（与 `validate:manifest`/`validate:structure` 同级）。
- `.husky/pre-push`（项目 `core.hooksPath=.husky`）追加 `npx tsx scripts/validate-rules-registry.ts`（链到现有 hook；若该 hook 文件不存在则按 husky 规范创建，含 `validate:manifest`+`validate:structure`+`validate:rules-registry` 三链）。

**L2 /qa E2E checklist**：加「点 revert/完成/取消 按钮 → 状态正确迁移」检查项（回归守护）。

**关闭条件**：TD-019 README 标 resolved，A1/A2 落地。

### §2 回退 Bug 验证（与 §1 L2 合并）

### §3 /timeboxes 打卡专区 + 快捷打卡

**数据模型（P2）**——`usom/types/objects.ts` 扩展：
```ts
export interface DetailedExecutionRecord extends ExecutionRecordBase {
  mode: 'detailed'
  completionRating: number
  actualOutput: string
  deviationReasons?: string
  energyLevel?: number
  notes?: string                    // 复用：任务执行详情（长文本）
  // [023.13] 新增
  actualStartTime?: Timestamp       // 实际开始时间
  actualEndTime?: Timestamp         // 实际结束时间（与 actualStartTime 派生 actualDuration）
  focusMinutes?: number             // 深度专注时长（rule: <= actualDuration）
  energyActual?: number             // 实际能量消耗 1-10（单值度量，默认 archetype 4 维均值）
}
```
- **这些是新字段，不是"恢复 [023.12] 移除的字段"（issue 3）**：`actualStartTime/actualEndTime` 存在 `execution_record` JSONB 里（ExecutionRecord 的子字段），**不是** `timeboxes` 表的列，也与 [023.12] 从未移除过的 `Timebox.startedAt/endedAt`（仍存在于 USOM/schema）无关。免 DDL 列迁移的根据是"JSONB 形状演进"，不是"恢复撤列"。
- **actualDuration 派生规则（issue 5）**：`actualDuration = round((actualEndTime - actualStartTime) / 60000)` 分钟。**仅当 actualStartTime 与 actualEndTime 同时存在**才派生并覆盖 base 的 actualDuration；只有 focusMinutes 无时间窗时，actualDuration 不派生（focusMinutes 独立校验 `<= actualDuration`，若 actualDuration 缺失则该校验跳过）。
- **能量默认值（issue 5）**：`getDefaultEnergyActual(archetype)` = `round((physical+mental+emotional+creative)/4)` 算术均值（本轮最简，加权留个性化模型版本）。**无 archetype**（`activityArchetypeId` 为空）：`energyActual` 默认 undefined，UI 强制用户手填（不默认 0，避免假数据）。
- mapper 向后兼容：旧记录新字段 undefined；规则引擎/UI 读 `focusMinutes`/`energyActual`/`actualStartTime`/`actualEndTime` 必须空值检查。

**按钮语义（P4）**——`timebox-card.tsx` 重排：
- planned 卡：`[✓ 一键打卡]`（quick simple log）`[打卡]`（开抽屉详细）`[取消]` `[删除]`
- 列表顶栏：`[多选]` toggle → 批量模式（每卡 checkbox + 底栏 `[批量打卡] [批量取消]`）
- logged/cancelled 卡：`[回退]`（logged 走确认弹窗）+ logged 增强现有 `[查看记录]`（issue 6：该按钮 `timebox-card.tsx:189-191/265-267` 已存在，**非新建**——本轮增强其 modal 渲染新增 5 字段，不是新交付）。

**回退（P3）**——`app/actions/timebox.ts:184-200` + repository + workspace（issue 4 精确化，plan 期修正 archive 误用）：
- ⚠️ **plan 期发现 `archive()` 不能清 executionRecord**：`repository/index.ts:114-125` 的 `archive(id,userId,executionRecord?)` 用 `if (executionRecord)` 真值判断——传 null/undefined 跳过写、**不清空**旧值，且会把 status 盖成 `'logged'`（语义错误，revert 该回 planned）。revertTransition 的 JSDoc（`repository/index.ts:127-133`）明示「彻底清 executionRecord 由调用点显式做，仓储不耦合 AM7」。
- **[eng-review AM3 精化] 不新增 clearExecutionRecord 方法**——复用现有 `repository/index.ts:99-112` 的通用 `updateFields(id, fields, userId, tx)`：`await repo.updateFields(id, { executionRecord: null }, userId, tx)`。单条 UPDATE + T-02 userId 过滤,与持久化修复(AM1)同通道,免新抽象(DRY)。
- `revertTimebox(id, { clearExecutionRecord?: boolean })` 分支：
  ```
  if (clearExecutionRecord) {
    await repo.updateFields(id, { executionRecord: null }, userId)   // 复用 updateFields(AM3)
  } else if (tb.executionRecord != null) {
    throw new Error('请先清理执行记录再回退')        // AM7 守卫保留（默认路径不变）
  }
  await submitDynamicIntent('timebox', 'revertTimebox', { objectId: id })
  ```
- **AM7 测试更新**：现有「executionRecord != null → 抛 AM7」测试改为「未传 clearExecutionRecord → 抛 AM7」；新增「传 clearExecutionRecord=true → archive 清空 + revert 成功 + executionRecord===null」测试。
- 前端：logged 卡点回退 → AlertDialog「将清除执行记录（实际时长/专注/能量/详情），不可恢复」→ 确认 → 带 `clearExecutionRecord=true` 调用。
- cancelled 卡点回退 → 直接调用（无弹窗，executionRecord 恒 null）。
- **appointment 回退不在本轮 P3 范围（issue 11）**：`revertAppointment`（`timebox.ts:486-508`）当前无 executionRecord 守卫（appointment lifecycle 无 executionRecord 概念，用的是 scheduledAt/completedAt/cancelledAt 时间戳），数据丢失语义不同。本轮 timebox-only；appointment 回退确认 UX 留 OQ-4。

**打卡抽屉**——`TimeboxDrawer` 新增「打卡专区」分区（计划信息保留在上），字段由共享组件 `ExecutionDetailFields`（见 §4）渲染：
- 实际开始/结束时间（datetime-local，两者齐备时自动算并显示实际时长）
- 深度专注时长（number，rule: `<= 实际时长`，actualDuration 缺失时跳过校验，超限红字）
- 实际能量消耗（number 1-10，有 archetype 默认 `getDefaultEnergyActual` 均值可调；无 archetype 留空强制手填）
- 任务执行详情（textarea → `notes`）

### §4 /logTimebox 共享 + 打卡专区

- 抽 `components/ExecutionDetailFields.tsx` 共享子组件（5 字段 + 实际时长派生 + 专注校验 + 能量默认）。
- /timeboxes 打卡抽屉 + /logTimebox CNUI 都消费它，**避免表单分叉**（memory: CNUI 表单分叉技术债）。
- **/logTimebox 分页 × 详细展开（issue 10）**：现 `LogTimebox.tsx` 是多 item 分页（`useState(0)` page index）。每个 item 独立持有 `detailedOpen` 标志——默认 simple（三态 + 备注），点「详细」展开实例化一份 `ExecutionDetailFields`，提交时该 item 升级为 `DetailedExecutionRecord`，未展开的仍 `SimpleExecutionRecord`。
- handler: `logTimebox` intent 接受 optional detailed 字段，落 `DetailedExecutionRecord`。

### §5 月历翻页

- `mini-calendar.tsx`：加内部 `viewMonth` state（`useState(() => startOfMonth(currentDate))`）+ `userTouchedMonth` 标志。
- 标题栏加 `‹ ›` 按钮，步进 viewMonth（`addMonths(±1)`）并置 `userTouchedMonth=true`。
- **同步规则（issue 9，二选一定死，T9 test-aligned）**：`currentDate` prop 变化时，**仅当 currentDate 自己从旧月份跨入新月份**（用 `useRef` 记前值 `prevCurrentMonthRef`，比较 `!isSameMonth(prev, current)`）时才重置跟随并清 `userTouchedMonth`；同月内 currentDate 变化不抢回 viewMonth。即：用户手动翻 → 锁定；currentDate 跨月 → 跟随；DateNav 切日不跨月 → 保持用户当前视图。（`!userTouchedMonth` 路径仅保留在首次 mount 同步默认 viewMonth。）
- `onDateSelect` 不变（点日期仍切 currentDate + 回 day view）。events 标点基于 viewMonth 网格。

## What I noticed about how you think

- 三个架构决策你全选了**推荐项**（单值能量 / 确认弹窗 / 全做），但每个推荐都是「不破既有决议的最小演进」——你没有为了"完整"去碰 D8 宪章 amendment，也没为了"顺滑"去推翻 AM7。这是在已知技术债红线内取最大价值的判断。
- 你把「打卡专区 4 字段」直接列进需求，但没意识到其中 3 个击穿了现行 schema——这恰恰说明你是从**用户场景**（我想记录什么）而不是**数据模型**（字段往哪塞）出发思考的。office-hours 的工作就是把这层翻译显式化。
- 需求文档里 §1（TD 预防）和 §2（回退 bug）你分开列了，但它们根因同一——这种"看起来两件事其实是一件"的合并，是 design 阶段该做的减法。
