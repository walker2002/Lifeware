# Design: [023.12] 三域生命周期语义重构（timebox / OKR cycle / appointment）

Generated by /office-hours on 2026-07-06
Branch: main
Repo: lifeware
Status: APPROVED
Mode: Builder

## Problem Statement

当前三个域的生命周期混淆了两类状态：

1. **用户行为状态**（应持久化）：用户主动动作产生的状态跃迁。
2. **自然时间状态**（不应持久化）：由 `now` 推导、与用户行为无关。

混淆导致：
- **timebox** 把 `running/overtime/ended` 写进 DB（start/overtime/end 动作 + 时间戳字段 + 4 个事件 + overlap rule 状态判断），但这些都是时间派生的——真正属于用户行为的只有"打卡"和"取消"。
- **okr cycle** 用 `not_started/in_progress` 这类时间感状态，实际只有"审批/结束/复盘"是用户行为。
- **appointment** 靠 lazy reconcile 把 `in_progress/expired` 写回 DB（每页加载逐条 `submitDynamicIntent`），把时间状态持久化的成本（SM 往返、写放大）付给了不该持久化的东西。

目标：把三个域的生命周期收敛到**只跟踪用户行为**，时间状态一律读时派生显示；同时调整三页 UI（`/timeboxes` `/okrs` `/appointments`）的操作按钮以匹配新语义。

## What Makes This Cool

"状态机里只放用户行为"——这是一个把领域模型从"模仿物理时间"收敛到"跟踪人类意图"的清理。一旦做完，三个域的 lifecycle 都变得可解释（每个 transition 都对应一个用户动作 + 一个事件），而"进行中/已超时/已过期"变成纯展示层关注点（共享一个派生工具）。这是对"状态机到底该建模什么"的一次正本清源。

## Constraints

- **manifest.yaml 是 lifecycle SSOT**（constitution G-01~G-08）；`lifecycle-configs.ts` 动态从 manifest 读取，所以主要改 manifest，不动 lifecycle-configs 的动态加载逻辑（但内联的 `@deprecated timeboxLifecycle` 常量要同步）。
- **Repository Pattern + 多租户**（R-01~R-04 / T-01~T-04）：状态写走 SM → repository.updateStatus，不绕过。
- **CNUI surface 双层注册**（[[project-cnui-surface-dual-registration]]）：新增/改 surface 须 server `surfaceHandlers` + client `register-client-surfaces` + manifest K-block + intent_trigger 四路闭合。
- **手写迁移规范**（[[project-drizzle-migrations-handwritten]]）：`db:generate/migrate` 跑不通，迁移一律手写 SQL + psql + 登记 journal（idx 续编）。
- **Tier 2 文档同步**（[[feedback_tier2-sync]]）：USOM/DB 变更先更新 `docs/usom-design.md` + `docs/database-design.md` + `CHANGELOG.md`。
- **注释规范**：每个改动的 TS/JS 文件保持 `/** @file ... @brief ... */` 头注释同步。
- **PG enum 不可 DROP VALUE**——但本项目无生产数据（见 Recommended Approach），此约束实际失效。

## Premises

**P1（timebox 核心）**：`running/overtime/ended` 不再持久化，UI 读时用 `now` vs `startTime/endTime` 派生。SM 只留 `null→planned`、`planned→logged(打卡)`、`planned→cancelled(取消)`，外加 `logged→planned`、`cancelled→planned` 两条回退。`startedAt/endedAt/overtimeAt` 字段 + `TimeboxStarted/Ended/Overtime` 事件 + `list_actions` 的 `start/end` 项一并清理。

**P2（okr cycle）**：`not_started/in_progress` 折叠为 `approved`，`ended` 改名 `finished`，加 `reviewed→finished` 回退。`draft/approved/finished` 可任意编辑；`reviewed` 不可删但可回退；cycle 下挂 objective 且 objective 有任务/习惯贡献时不允许删 cycle。`objective.ts` 现按 `cycle.status='in_progress'` 过滤"活跃 objective"，改为 `'approved'`。**注意（reviewer #1）**：本轮**不只是改 status guard**——cycle 的 SM action 名 + server action 分派 + guard ALLOWED map + cycle.ts 时间戳分支都要重命名（现 `startCycle/planCycle` 二选一分支要塌缩成单一 `approve`），详见 §Detailed/cycle 映射表。

**P3（appointment）**：`in_progress/expired` 移除，改读时派生显示（D1 决议，同 timebox 模式）。`completed` 正式启用（`scheduled→completed`）。回退 `cancelled/completed→scheduled`（D2 决议）。`scheduled→cancelled`/`completed` 在有任务/习惯关联时拦截。

**P4（DB 迁移现实）**：三表 `status` 列是 **plain TEXT**（drizzle `text({ enum })` 仅 app 层 union，无 PG enum type，见 §迁移方案 更正）——不存在"PG enum DROP VALUE 不可"约束。现有数据全为测试、生产库未录入（D3 决议）→ TRUNCATE 清旧值 + 改 schema.ts `enum: [...]` 数组即可，零行迁移、零 DDL on status、零风险。

**P5（节奏）**：三域共享 lifecycle 基础设施（manifest/lifecycle-configs/SM），一次 ship 最省事（D3 决议 = 方案 A）。

## Approaches Considered

### Approach A: 三域同船 + enum 增量（死值）
1 分支改完三域；PG enum 只 ADD 新值，旧值保留为死值（迁移行后不再写入）。Coherent 但 enum 留尸骸。

### Approach B: 分域三 PR + enum 增量
timebox → okr → appointment 逐域 ship。爆炸半径小，但 lifecycle 共享层反复改、合并冲突高、仪式 ×3。

### Approach C: 三域同船 + enum 重建类型（理想）
同 A 范围，PG enum 用重建类型迁移。schema 最干净，但迁移 SQL 复杂、prod 锁表风险最大。

> **ERRATUM（writing-plans 阶段）**：方案 A/B/C 三选一的整个"PG enum 死值 vs 重建"维度基于错误前提——三表 status 列实测是 plain TEXT（无 PG enum type，见 §迁移方案 更正）。C 出局不是因为"风险大"，而是因为**没有 enum 可重建**。最终落地 = 方案 A 的范围 + 零类型 DDL。

## Recommended Approach

**方案 A（1 ship / 三域同船）**——单分支改完三域，lifecycle 共享层（manifest/lifecycle-configs/SM）一次到位。迁移比预期更简（§迁移方案 更正）：status 列是 TEXT 不是 PG enum，"移除旧值"只是改 schema.ts `enum: [...]` 数组 + TRUNCATE 清行 + DROP 废弃时间戳列，零类型重建、零 prod 风险。方案 C 的"enum 重建类型"基于一个不存在的前提（误以为有 PG enum），直接出局。

理由映射目标：lifecycle 基础设施跨域共享，分开改（方案 B）重复仪式且合并冲突高；"读时派生"做成跨 timebox+appointment 的共享派生工具，是本轮最值得抽的一层。

### 与 [026] D2 reversal 的关系（plan-eng-review codex #1 补）

`reconcile-appointment.ts` 文件头明载：[026] D2 reversal **取代**了原 D2=C "读时算 status" 派生方案，改为"状态全部存 DB + lazy reconcile 写库"。本轮 [023.12] 恰好退回 [026] 推翻的读时派生模型。**这是有意的、显式的反转**，理据如下：

- [026] 选持久化的关键动机是"D2=C 读时派生无法 enforce 写入约束（expired 不可删）、无法支撑'未实施计划'统计"。本轮重新评估后认为：写入约束（不可删/不可改）可以由 **SM 持久态**（cancelled/completed/scheduled）单独 enforce，无需把"时间态"也持久化；"未实施计划"统计 = `status='scheduled' 且派生 expired`，派生函数足以回答。
- [026] 的 lazy reconcile 路径（每页加载 N 条 submitDynamicIntent 写库）引入写放大 + SM 往返成本，单用户 MVP 不值得。
- **反转的代价已被接受**：派生后 `currentTimebox`（orchestrator 写入的 JSONB）改由 derive 填充（见 plan T13），overlap 规则改应用层派生（见 plan T7 Step 1b），SQL 级"查过期约定"不再可能（见下方 trade-off）。

**Trade-off（plan-eng-review codex #11）**：时间派生态（running/overtime/in_progress/expired）不再可 SQL 查询——"SELECT 过期约定""COUNT 运行中时间盒"须加载 scheduled/planned 行后应用层派生过滤。单用户 MVP 可接受；若未来引入报表/分析需求，需在那时重新持久化（或物化视图）。

### 与 codex "拆分行为 vs schema 清理" 建议的关系

plan-eng-review codex #2 建议拆分：(A) 停止持久化时间态 + 派生显示，(B) schema enum/列清理——本轮只做 (A)，(B) 降 follow-up。**用户经评审后选择不拆分**（评审 D-shape → B）：要一次到位的干净 schema（旧值直接删）。代价是处理 codex #3 currentTimebox / #4 overlap 算法 / #6 TZ / #8 cycle 改名 + 补齐漏网文件——这些已折入 plan amendments（见 plan-eng-review amendments 段）。

## Detailed Lifecycle Definitions

### timebox（持久态：planned / logged / cancelled）
```yaml
states: [planned, logged, cancelled]
initial_state: planned
transitions:
  - { from: null,      to: planned,   action: create, event_type: TimeboxCreated }   # 新建
  - { from: planned,   to: logged,    action: log,    event_type: TimeboxLogged }     # 打卡
  - { from: planned,   to: cancelled, action: cancel, event_type: TimeboxCancelled }  # 取消
  - { from: logged,    to: planned,   action: revert, event_type: TimeboxReverted }   # 回退（新）
  - { from: cancelled, to: planned,   action: revert, event_type: TimeboxReverted }   # 回退（新）
terminal_states: []  # 无终态——logged/cancelled 都可回退；改用"可修改性规则"替代
```
- **可修改性规则**：planned 可编辑/可删；logged/cancelled 不可编辑/不可删，但可回退。
- **派生显示状态（不持久化）**：`running` = planned 且 `now ∈ [startTime,endTime]`；`overtime` = planned 且 `now > endTime`。
- **字段清理**：移除 `startedAt/endedAt/overtimeAt`（lifecycle_timestamp）三列 + 对应 field_metadata。
- **事件清理**：移除 `TimeboxStarted/TimeboxEnded/TimeboxOvertime`；新增 `TimeboxReverted`。
- **list_actions**：收敛为 `[log(打卡), cancel(取消), revert(回退), delete(删除)]`，移除 `start/end`。

### okr cycle（持久态：draft / approved / finished / reviewed）
```yaml
states: [draft, approved, finished, reviewed]
initial_state: draft
transitions:
  - { from: null,       to: draft,    action: create,  event_type: CycleCreated }    # 新建
  - { from: draft,      to: approved, action: approve, event_type: CycleApproved }   # 审批
  - { from: approved,   to: finished, action: finish,  event_type: CycleFinished }   # 结束
  - { from: finished,   to: reviewed, action: review,  event_type: CycleReviewed }    # 复盘
  - { from: reviewed,   to: finished, action: revert,  event_type: CycleReverted }    # 回退（新）
terminal_states: []
```
- **可修改性规则**：draft/approved/finished cycle 及其 objective/KR 可任意编辑；reviewed cycle 锁定（objective/KR 写路径 guard）。
- **删除守卫**：cycle 下有 objective（且 objective 有任务/习惯贡献 junction）时不允许删；reviewed cycle 不允许删（可回退到 finished 后再删）。
- **活跃 objective 过滤**：`cycle.status === 'approved'`（替换原 `'in_progress'`）；okr-directory tabs = `draft/approved/finished/reviewed`；默认展开 cycle = approved。
- **现有 cycle-menu 三 action（approveCycle/endCycle/reviewCycle）**：本轮**不止改 status guard**——SM action 名 + server action 分派 + guard ALLOWED map 都要重命名。映射表：

| 现状（server action → SM path） | 新模型（SM action） | 说明 |
|---|---|---|
| `approveCycle` → getAction `startCycle \| planCycle`（按 now vs periodStart 分支） | `approve`（intent `approveCycle`） | 塌缩二选一分支；draft→approved 单一转换 |
| `endCycle` → `'endCycle'` | `finish`（intent `finishCycle`） | rename；approved→finished |
| `reviewCycle` → `'reviewCycle'` | `review`（intent `reviewCycle`） | 语义不变；finished→reviewed |
| （不存在） | `revert`（intent `revertCycle`） | 新增；reviewed→finished |
| manifest actions: `plan/start/end/review` | manifest actions: `approve/finish/review/revert` | block B transitions 重写 |
| `guard.ts` ALLOWED keys: `draft/not_started/in_progress/ended/reviewed` | keys: `draft/approved/finished/reviewed` | draft/approved/finished = `{edit_objective, edit_kr}`；reviewed = `{}` |
| `cycle.ts` 时间戳: `in_progress→startedAt, ended→endedAt, reviewed→reviewedAt` | `approved→startedAt, finished→endedAt, reviewed→reviewedAt` | updateStatus 分支 rename |

- **同源 status 字面量散落点（reviewer #2，全域 grep）**：`objective.ts:101`（过滤）、`okr-directory.tsx:71-73`（tabs 数组）+ `:124`（hasActive 折叠判定）、`guard.ts:29-32`（ALLOWED）、`contribution-panel.tsx:70`（`cycleStatus !== 'reviewed'`——**幸存不变**，reviewed 语义保留）、相关 vitest fixture（`guard.test.ts`、`contribution-panel.test.ts`）。
- objective / key_result 的单态 lifecycle（`[draft]`，null→draft）**不动**——`objectives`/`key_results` 表 status 列不受 0034 迁移影响。

### appointment（持久态：scheduled / cancelled / completed）
```yaml
states: [scheduled, cancelled, completed]
initial_state: scheduled
transitions:
  - { from: null,       to: scheduled, action: create,   event_type: AppointmentCreated }    # 新建
  - { from: scheduled,  to: cancelled, action: cancel,    event_type: AppointmentCancelled }  # 取消
  - { from: scheduled,  to: completed, action: complete,  event_type: AppointmentCompleted }  # 完成（启用，原 [027] 注释）
  - { from: cancelled,  to: scheduled, action: revert,    event_type: AppointmentReverted }   # 回退（新）
  - { from: completed,  to: scheduled, action: revert,    event_type: AppointmentReverted }   # 回退（新）
terminal_states: []
```
- **可修改性规则**：scheduled 可任意编辑/可删；cancelled/completed 不可编辑/不可删，但可回退。
- **取消/完成守卫**：需求要求"有任务/习惯关联时不允许 cancel/complete"，但 schema 无 appointment↔task/habit junction（OQ-1）→ **本轮降级 TODO**，按钮先无条件放行，留 `// TODO [027]: appointment task/habit guard`。
- **派生显示状态（不持久化）**：`in_progress` = scheduled 且 `now` 与 `startTime` 同日历日；`expired` = scheduled 且 `now` 日历日 > `startTime` 日历日。
- **reconcile 文件归因修正（reviewer #3）**：现状是两个文件——
  - `status/reconcile-appointment.ts`（singular）**已经是纯函数**（返回 `ReconcileAction[]`，不 IO 不写库）→ 本轮**改造为 badge 派生函数**（输入 `Appointment[]`+`now`，输出 `{appointmentId, badge: 'in_progress'|'expired'|null}[]`）。
  - `app/actions/reconcile-appointments.ts`（plural）才是写库入口（直接走 repo path，不经 submitDynamicIntent），由 `app/appointments/page.tsx:21` 的 `reconcileAndAdvanceAppointments(MVP_USER_ID)` 调用 → 本轮**删除该 action（或导出空 no-op + deprecated 注释）+ 删除 page.tsx:21 调用**。
  - `domains/timebox/pages/AppointmentPage.tsx` 注释也提及 reconcile，同步清理。

## 迁移方案（数据可弃 → status 是 TEXT，无 PG enum）

**关键事实更正（writing-plans 阶段实地核对 schema.ts + 0001 迁移注释）**：三表 `status` 列都是 **plain `TEXT`**，drizzle 的 `text('status', { enum: [...] })` 只在 TS 层做 union 约束，**DB 层无 PG enum type**（`0001_execution_logging.sql:5` 注释明写"status 是 TEXT 类型，无需 ALTER ENUM"；schema.ts grep 无 `pgEnum`）。因此：
- **不存在** "PG enum DROP VALUE 不可" 的约束（P4 前提更正）。
- **不需要** DROP TYPE / CREATE TYPE / 重建类型（方案 A vs C 的整个辩论基于错误前提，C 的"复杂迁移 SQL + prod 锁表风险"不存在）。
- "移除旧值" = 改 schema.ts 的 `enum: [...]` 数组（纯 app 层，零 DDL）+ 清掉含旧值的行。

迁移 0034 实际只做两件事：**TRUNCATE（数据可弃，免行迁移）+ DROP 废弃列**。

新增 `frontend/src/lib/db/migrations/0034_023_12_lifecycle_simplify.sql` + 登记 journal（idx=34，按 [[project-drizzle-migrations-handwritten]] 规范）：

```sql
-- timeboxes: drop 3 个时间戳列（status 列保留为 TEXT，旧值随 TRUNCATE 清掉）
TRUNCATE timeboxes CASCADE;
ALTER TABLE timeboxes DROP COLUMN IF EXISTS started_at;
ALTER TABLE timeboxes DROP COLUMN IF EXISTS ended_at;
ALTER TABLE timeboxes DROP COLUMN IF EXISTS overtime_at;

-- cycles: status 列保留 TEXT（无废弃列），仅 TRUNCATE 清旧值
TRUNCATE cycles CASCADE;

-- appointments: drop 2 个时间戳列
TRUNCATE appointments CASCADE;
ALTER TABLE appointments DROP COLUMN IF EXISTS in_progress_at;
ALTER TABLE appointments DROP COLUMN IF EXISTS expired_at;
```
- status 列的"合法值集合"由 schema.ts 的 `enum: [...]` 数组在 app 层约束，本轮改数组即可（Task 2），**无 DDL**。
- **CASCADE 影响表（reviewer #4，已核对 schema.ts FK）**：
  - `TRUNCATE timeboxes CASCADE` → 清空 `task_timeboxes`（CASCADE junction）、`timebox_habits`（CASCADE junction）；`task_execution_logs.timebox_id` SET NULL（行留，列置空）；**不动** objectives/cycles（无 FK 到 timeboxes）。
  - `TRUNCATE cycles CASCADE` → 清空 `objectives`（`cycleId` 虽 RESTRICT，但 TRUNCATE CASCADE 绕过 restrict 校验仍清空）→ 级联清空 `key_results`（FK to objectives CASCADE）。
  - `TRUNCATE appointments CASCADE` → 无 FK 反向依赖（schema grep 0 hits），CASCADE 实际 no-op。
  - **`memory_episodes.metadata` jsonb 软引用**：[023.08] F4 batch undo 通过 EpisodeRepository 把 timeboxId 写入 memory_episodes.metadata，TRUNCATE timeboxes 不动该表 → 留孤儿软引用。决议：**接受孤儿**（无 FK、无读路径反向解析 timeboxId，undo 已 ship 的 batch 仅作审计痕迹），不一并 truncate memory_episodes。
- `0034_..._down.sql` 反向重建废弃列（prod 不需要，dev 回滚兜底）。
- seed 脚本（如有依赖）同步更新。
