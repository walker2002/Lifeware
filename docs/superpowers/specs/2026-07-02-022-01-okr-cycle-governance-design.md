## 问题陈述

[022] Phase 1 落地了 Cycle 一级对象，但留下一批治理债，本次集中正规化：

1. **Cycle 写入违宪**：`createCycle` / `findOrCreateCycle` 直写 `repo.save` / `tx.insert` 绕过 SM 与业务事实写入口（违宪章 §III「业务事实写入口」+ §IX.1「写入口两条合法路径」；`status` 是 FactField，§III How-to-apply 明确「直接写 status 字段绕过 SM」须拒）。`write-entry-guard` 的 allow 例外缺 sunset 且理由覆盖错层，不合规。
2. **Cycle 生命周期未实现**：manifest 定义了 5 态，但无代码做 start/end/review；not_started/ended/reviewed 是死状态；无时间驱动。
3. **Objective/KR 独立状态机与 Cycle 脱耦**：Obj/KR 各自一套 lifecycle，与所属 Cycle 状态不联动（如「导入建 draft cycle → 目标不在 in_progress 列表 → 全不可见」黑洞，[024.2] 临时修复）。
4. **目录筛选错位**：左侧目录按 Objective 状态筛选，权威应是 Cycle 状态。
5. **模板导入不提取周期**：模板无 Cycle 信息，导入建匿名 cycle。

## P0 决策（spec review iteration 2 拍板，不再推迟）

### P0-1：实现 `adapter.cycle.create`（当前抛错）—— 让 executeIntent 能创建 cycle
- **现状（实测）**：`generic-repo-adapter.ts` L153-155 `cycle.create` **直接抛错** `throw new Error('Cycle 不支持通过 GenericRepo 创建，请使用 CycleRepository')`；`cycle.updateStatus` 也抛错。这是 createCycle/findOrCreateCycle 绕过 SM 直写 `repo.save` / `tx.insert` 的**根因**——executeIntent 无 cycle.create 可用，只能 bypass。
- **决策（Phase 1）**：实现 `adapter.cycle.create(fields, userId, tx)`：按 fields（cycleType/name/periodStart/periodEnd）+ manifest initial_state `draft` + 新生成 id/时间戳构造 Cycle → `return await repos.cycleRepo.save(cycle, userId, tx)`（save 是 onConflictDoUpdate + 自然键回查，返回持久化行含真实 id）。这样 executeIntent create → SM `null→draft` → adapter.cycle.create → save，整链合规且幂等。
- **幂等保障**（三重）：SM `null→draft` 是初始转换（重复触发在 SM 层由 from-state 守护）；`CycleRepository.save` 的 onConflictDoUpdate（cycle.ts L44，自然键 userId+periodStart+periodEnd）保证同自然键二次提交不报错且返回同一行；findOrCreateCycle 保留前置 select（L130-132）作快速路径与额外幂等闸。补单测：重复调用 `findOrCreateCycle`（经 SM）返回同一 cycle.id；同自然键二次 executeIntent create 不报错。

### P0-2：time-trigger 的写入路径 —— 走 `createDomainMutationService.execute(intent)`（单 state 步骤 AggregateIntent），不经 Intent Engine
- **问题**：orchestrator 公开写入入口只有 `executeIntent`（经 Intent Engine）；L390 要求「All external writes MUST traverse the full Nexus chain (Intent Engine → Rule Engine → 写入口)」。但 L145 明文「SM accepts only StateProposal from Orchestrator or **time triggers**」——time trigger 是与 Orchestrator 平行的合法 StateProposal 发送方，是 L390 的**显式例外**。§IX.1 列两条合法写路径：(1) `executeIntent`，(2) `createDomainMutationService.{update,execute}`。time-trigger 用路径 (2)。
- **决策**：`sweepCycles` 构造一个**仅含单 state 步骤的 AggregateIntent**（MutationStep 走 state 路径，from→to=start/end），经 `createDomainMutationService.execute(intent, userId)` 提交 SM。**不**经 Intent Engine、**不**构造 stub intent。依据：L145（time trigger 豁免 Intent Engine）+ §IX.1 路径 (2)。无需宪章修订，无需扩 mutation-service 签名（AggregateIntent 本就是 execute 的合法入参）。

## 推荐方案（架构总览）

### A. Cycle 写入口正规化（Phase 1）

- `createCycle`（`app/actions/okr.ts`）：改走 `executeIntent(makeIntent("createObjective"... 错，应为 "create" cycle intent))`。manifest cycle 的 create action 是 `create` → `draft`。makeIntent 构造 `{ targetDomain: "okrs", action: "create", fields: { cycleType, name, periodStart, periodEnd } }`（cycle 在 adapter 由 objectType=cycle 路由）。**不再接受 status 入参**（强制 draft）。
- `findOrCreateCycle`（`app/actions/okr-import.ts`）：改走 executeIntent create。自然键幂等由「P0-1 改造后的 adapter.create 返回 save 回查行」+ CycleRepository.save 的 onConflictDoUpdate 共同保证。**去掉 status 参数**（import 建的 cycle 也是 draft，与 [024.2] 的 in_progress 临时硬编码冲突 —— 退役它，见「数据迁移计划」）。
- `write-entry-guard.test.ts`：**删除** createCycle 的 allow 例外；**守卫扩展仅作用于 `FILES = ['okr.ts','okr-import.ts']`**（per-file，不扩到全库，避免误伤其他域合法 .insert），新增 grep `.insert(s.cycles)`（仅 cycle 表，精准拦截 findOrCreateCycle 类直写，不波及其他合法 .insert）。合法的 cycle 写一律走 executeIntent / adapter。
- `generic-repo-adapter.ts` cycle.create：**实现**（当前抛错），见 P0-1。adapter.cycle.updateStatus 同步评估是否需实现（Phase 2 cycle 状态转换要用，至少 start/plan/end/review）。
- `CycleCreateDrawer`：去掉 `status: "in_progress"` 硬编码；构造 Cycle 时传 `status: 'draft'` 占位（USOM Cycle 类型仍要 status 字段），SM 按 manifest `create → draft` 强制 draft。UI 提示「建完需审核通过后启动」。

### B. 移除 Objective/KR 独立状态（Phase 3）

见 P0-3 清单（组 1-4）。核心：Obj/KR 无 status 字段、无 SM、无 lifecycle；编辑/删除权限完全由所属 cycle.status 经 `assertEditable` 守卫；完成语义由 KR 进度承载；软删除复用 discardedAt/archivedAt。

### C. Cycle 状态权限模型（Phase 2）

新增 `assertEditable(cycleId, userId, operation: "edit"|"delete")` 于 okrs mutation-service 层，所有 Cycle/Obj/KR 写入口调用。**Obj/KR 自身无 status，可编辑性 = 字段写许可，由 cycle.status 决定，不矛盾**：

| Cycle 状态 | Cycle 改 | Cycle 删 | Obj/KR 字段改 | Obj/KR 删 |
|---|---|---|---|---|
| draft | ✅ | ✅（有目标禁） | ✅（不记日志） | ✅（关联守卫：有 KR 的 Obj / 有 contribution 的 KR 禁删） |
| not_started / in_progress / ended | ❌ | ❌ | ✅（写 audit log） | ❌ |
| reviewed | ❌ | ❌ | ❌ | ❌ |

### D. Cycle UI 操作（Phase 1/2）

- **审核通过**（Cycle ⋯ 菜单，仅 draft 显示）→ 弹窗二次确认 → 按 server 时刻 `now`（UTC，与 periodStart/periodEnd 同为 date-only 比较到日）分派：
  - `now >= periodStart` → executeIntent `start`（draft → in_progress）
  - `now < periodStart` → executeIntent `plan`（draft → not_started）
- **删除**（仅 draft 且无目标）→ 弹窗二次确认 → deleteCycle。
- **复盘**（仅 ended，可选）→ executeIntent `review`（ended → reviewed）。
- 顶部筛选 tabs 改 Cycle 状态：全部 / draft / not_started / in_progress / ended / reviewed。
- **manifest 新增转换**（Phase 1）：`draft → in_progress`（action: start, trigger: intent, event: CycleStarted）。**只新增这一条**（现有已有 `draft → not_started (plan)` 与 `not_started → in_progress (start)`）。manifest `trigger` 字段支持 `intent | time | intent+time`（见 P0-2 / §E）。

### E. 时间驱动 读时懒扫描（Phase 4）

- `sweepCycles(userId)` server action：
  - 查询候选 cycle：`(status='not_started' AND currentDate >= periodStart)` 或 `(status='in_progress' AND currentDate > periodEnd)`。currentDate = server 当地 date-only（与 periodStart/periodEnd 同精度，避免时区/0:00 边界歧义）。
  - 每个候选构造单 state 步骤的 AggregateIntent（action=start|end）→ 经 `createDomainMutationService.execute(intent, userId)`（P0-2 决策的路径）→ SM 校验 from-state + 转换 + 发 CycleStarted/CycleEnded。
  - 幂等：SM 由 from-state 守护，重复调用无副作用；且查询本身只挑 status 未跟上的。
- 触发时机：`useOKRs` hook 挂载时 fire-and-forget 调一次（**注意**：虽在 hook 里调，但它是后台 SM 写操作，措辞为「工作台挂载时触发一次后台 sweep」，不是「读时派生」——status 是真落库的，只是触发时机借了首次加载）。
- manifest：`not_started → in_progress (start)` 标 `trigger: intent+time`（sweep 与人工都可触发）；`in_progress → ended (end)` 标 `trigger: intent+time`。

### F. 修改日志 audit（Phase 5）

- Phase 5 先确认/补齐 `user_audit_log` schema（user_id / object_type / object_id / field / old_value / new_value / changed_at / source）。
- Obj/KR 字段更新在 not_started/in_progress/ended 期间：mutation-service `updateFields` 同事务写一条 audit。
- **时序**：Phase 2 引入权限矩阵时，audit 表 schema 在 Phase 5 才确认 —— Phase 2-4 期间 audit 暂不写（assertEditable 守卫先就位，audit 接线在 Phase 5）。本设计明确此顺序，不要求 Phase 2 即写日志。

### G. OKR 模板调整（Phase 6，建议拆 6a/6b）

- 6a：模板 Markdown 顶部新增 Cycle 段（type/name/start/end）；`okrExportTemplatesToMarkdown` 输出；`SYSTEM_PROMPT` 增 Cycle 提取指令；`parseOKRMarkdown` 解析 → `ImportResult.cycle?`。
- 6b：导入流程——模板**有** cycle → executeIntent create cycle（draft）+ create objectives（draft）；模板**无** cycle → UI 阻塞弹窗（复用 CycleCreateDrawer 或 cycle picker），用户选/建 cycle 后才继续。

## 数据迁移计划（Phase 3，含 down 与校验）

**up SQL**（一条迁移，手写 + 登记 journal idx=N）：
```sql
-- 1. 软删除语义落地（COALESCE 保已有时间戳）
UPDATE objectives SET archived_at  = COALESCE(archived_at,  now()) WHERE status='archived';
UPDATE objectives SET discarded_at = COALESCE(discarded_at, now()) WHERE status='discarded';
UPDATE objectives SET completed_at = COALESCE(completed_at, now()) WHERE status='completed';
UPDATE key_results SET archived_at = COALESCE(archived_at, now()) WHERE status='archived';
-- 2. DROP 索引（基于 status 列）
DROP INDEX IF EXISTS idx_objectives_user_status;
DROP INDEX IF EXISTS idx_key_results_user_status;
-- 3. DROP 列
ALTER TABLE objectives  DROP COLUMN status;
ALTER TABLE key_results DROP COLUMN status;
```

**down SQL**（回滚用，登记 journal 同一 idx）：
```sql
ALTER TABLE key_results ADD COLUMN status text;  -- 实际类型对齐原 enum
ALTER TABLE objectives  ADD COLUMN status text;
CREATE INDEX idx_objectives_user_status  ON objectives(user_id, status);
CREATE INDEX idx_key_results_user_status ON key_results(user_id, status);
-- status 值无法精确还原（已丢），回滚后全 NULL，应用层须容忍
```

**迁移后这些行的归属与用户视角（C4）**：
- active(546)/draft(13)/paused(1) objective：no-op，status 消失后**它们继承所属 cycle 的状态约束**。dev 库大部分 cycle 是 in_progress → 这些 objective 仍可字段编辑（符合「in_progress 期间 Obj 可改」），**用户视角无突变**。若某 objective 属 reviewed cycle，则锁编辑（符合预期）。
- completed(2)：completedAt 已设，KR 进度 100%，用户视角「完成的 KR」依旧可见。
- archived(26)/discarded(19)：discardedAt/archivedAt 已设，`findAll` 过滤掉，用户视角「已归档/废弃」依旧不可见。

**[024.2] 退役时机（C5）**：Phase 1 同时改 `createCycle` 与 `findOrCreateCycle` 走 SM create → draft。`okr-import.ts` L177 的 `'in_progress'` 硬编码随之删除（import 建的 cycle 也是 draft，需用户审核通过）。即 Phase 1 退役 [024.2]。

**历史 cycle 状态（CS4）**：现有 8 个 in_progress cycle **保留现状**，不强制回归 draft。新模型下它们会被 sweepCycles 在 `currentDate > periodEnd` 时自动转 ended（属正确行为：已过期的 in_progress 应结束）；用户可手动 `review` 转 reviewed。历史 cycle 的「未审核」与新建 cycle 的「需审核」语义割裂，作为已知边界接受（文档说明，不追溯）。

**T-02 多租户**：所有迁移 WHERE 含 `user_id`（若有 UPDATE），本迁移为全表 DDL + 无 user 过滤的 UPDATE（status 是全局列），无多租户风险。
