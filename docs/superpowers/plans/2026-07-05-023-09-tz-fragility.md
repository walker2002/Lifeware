## Task 1: #I-3 orchestration-handler TZ fragility 治本

**Files:**
- Modify: `frontend/src/domains/timebox/handlers/orchestration-handler.ts` (line 207-218, line 314-326)
- Modify: `frontend/src/domains/timebox/handlers/__tests__/orchestration-handler.test.ts` (新 3 tests)

**Interfaces:**
- Consumes:
  - `TimeboxSummary[]` (既有 USOM type, `startTime: Timestamp` (= UTC ISO string))
  - `new Date(tb.startTime)` 创建本地 Date 对象 — `getUTCHours()` 直接读 UTC hour 数（24h 0-23），**不依赖** process TZ env
- Produces:
  - `extractOccupiedSlots` 返 `TimeSlot[]` with `startHour/endHour` 现为 UTC 0-23
  - `detectConflicts` 用 UTC hour arithmetic 与 UTC hour 字符串 format message (zone-consistent with `proposal.payload.startTime` 来自 `formatTime(cursorHour, ...)` cursorHour 为 UTC)

**背景 (root cause)**:
- DB schema 存 UTC ISO timestamp（`'2026-07-05T08:00:00Z'`）
- USOM handler 调 `new Date(tb.startTime)` 创建 Date 对象
- 但 `start.getHours()` 读浏览器 local TZ 的 hour（如 CST +08:00 浏览器读 `08:00:00Z` Date 返 `startHour=16`）
- 后果: `findOccupyingSlot` / `isSlotOccupied` / arithmetic compares 错位 hour → bound 误触发/误不触发, conflict detection 漏报/假报
- 修复: `.getHours()` → `.getUTCHours()` 直接读 UTC hour；与 DB 储存 canonical 一致；arithmetic 与 payload.startTime (UTC) zone-consistent
