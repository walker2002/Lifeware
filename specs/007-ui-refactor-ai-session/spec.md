## Clarifications

### Session 2026-05-16

- Q: LLM 未配置时 AI 助手标签页如何表现？ → A: 显示引导提示"请先配置大语言模型"，提供跳转至设置页的按钮，对话输入框隐藏
- Q: 会话归档与删除是同一操作还是不同操作？ → A: 两个独立操作。归档可恢复（从活动列表移除但数据保留）；删除不可逆（彻底清除消息数据）
- Q: 从 AI 对话视图切换到功能 action 时如何处理当前对话？ → A: 自动保存当前对话，主显示区切换到 action 视图，用户可通过左侧会话列表随时恢复对话

### Session 2026-05-17 (Clarify)

- Q: 非创建类 action（如 activateHabit, suspendHabit）无 required_fields 时，主显示区展示什么？ → A: 确认界面：展示目标对象摘要信息 + 确认/取消按钮
- Q: 动态表单/确认界面提交后的执行路径？ → A: 复用 Intent 路径，将动态表单数据映射为 StructuredIntent，通过现有 Intent Engine → onValidate → 状态机执行

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 通过快捷方式快速触发系统功能 (Priority: P1)

用户无需通过自然语言描述意图，直接输入快捷方式（如 `/createHabit`）即可快速跳转到对应功能的填写界面。系统识别快捷方式后，立即展示对应功能的表单或编辑区域。

**Why this priority**: 快捷方式解析是所有功能导航的基础，直接影响后续 UI 重构中菜单和对话的联动。独立交付即可让用户快速跳转到功能界面。

**Independent Test**: 在输入框中键入 `/createHabit`，验证系统跳转至习惯创建界面并展示对应表单。

**Acceptance Scenarios**:

1. **Given** 用户在任何界面，**When** 输入 `/createHabit` 并提交，**Then** 主显示区展示习惯创建表单
2. **Given** 用户输入了未注册的快捷方式 `/nonexistent`，**When** 提交，**Then** 系统提示该命令不存在，引导用户使用已有命令或自然语言描述
3. **Given** 两个功能领域定义了相同的快捷方式别名，**When** 系统启动，**Then** 系统拒绝启动并报告快捷方式冲突错误

---

### User Story 2 - AI 会话持久化与历史续接 (Priority: P1)

用户与 AI 助手的对话能够跨页面切换和跨会话保存。用户可以创建新对话、查看历史对话列表、继续之前的对话。继续旧对话时，AI 知晓对话期间相关对象的状态变化。

**Why this priority**: AI 会话是用户与系统交互的核心入口之一。会话持久化是后续 UI 重构中对话视图的基础。

**Independent Test**: 创建一个新对话并发送消息，切换到其他页面后再返回该对话，验证消息仍在且上下文完整。

**Acceptance Scenarios**:

1. **Given** 用户点击"新对话"，**When** 发送一条消息，**Then** 会话自动保存，左侧面板显示该会话条目（以首条消息摘要为标题）
2. **Given** 用户有多个历史会话，**When** 点击某个旧会话，**Then** 主显示区恢复该会话的全部消息记录
3. **Given** 用户在会话中引用了某个习惯对象，**When** 该习惯的状态在外部被修改后重新打开会话，**Then** AI 收到状态变更的系统消息，知晓习惯已从"草稿"变为"激活"
4. **Given** 用户点击"归档"一个会话，**When** 确认操作，**Then** 该会话从活动列表移除，可在"已归档"视图中查看和恢复
5. **Given** 用户对已归档会话点击"删除"，**When** 确认不可逆操作，**Then** 该会话及其全部消息被彻底清除

---

### User Story 3 - 新版三栏布局与导航体验 (Priority: P2)

用户看到全新的界面布局：左侧面板提供导航入口（AI助手会话列表、成长领域菜单），主显示区默认展示当天时间盒安排，AI 对话移入主显示区而非左侧面板。用户可通过左侧面板在不同功能视图之间切换。

**Why this priority**: UI 框架是后续功能接入（P4/P5/P6）的壳，需在功能接入前就位。

**Independent Test**: 启动应用，验证左侧面板包含 Home 按钮、AI助手标签、成长领域标签；主显示区默认显示时间盒。

**Acceptance Scenarios**:

1. **Given** 用户首次打开应用，**When** 页面加载完成，**Then** 左侧面板显示 Home 按钮和标签页，主显示区显示当天时间盒安排
2. **Given** 用户在任意视图，**When** 点击 Home 按钮，**Then** 主显示区回到当天时间盒视图，如果当前有活跃对话则自动保存
3. **Given** 用户在 AI 助手标签页，**When** 查看左侧面板，**Then** 仅显示会话列表（SessionList）和新对话按钮，不显示 AI对话/表单填写模式切换、输入框、确认弹窗等旧内容
4. **Given** 用户在主显示区查看时间盒安排，**When** 观察 UI，**Then** 仅显示 DateNav 和日历视图，不显示习惯库/模板/OKR/项目等标签页
5. **Given** 用户需要访问习惯库或 OKR 界面，**When** 切换到成长领域标签页并点击对应 action，**Then** 主显示区加载对应的功能界面

### User Story 4 - 成长领域菜单导航与功能触发 (Priority: P2)

用户通过左侧面板的"成长领域"标签页浏览所有可用的功能操作，按领域分组（习惯、OKR、项目/任务、时间盒）。点击任一操作项，主显示区加载对应的功能界面。

**Why this priority**: 功能菜单是用户发现和使用系统能力的主要入口之一，依赖 P1 的快捷方式注册和 P3 的 UI 框架。

**Independent Test**: 打开成长领域标签页，点击"创建习惯"，验证主显示区展示习惯创建表单。

**Acceptance Scenarios**:

1. **Given** 系统已注册习惯和 OKR 领域的多个 action，**When** 用户打开成长领域标签页，**Then** 菜单按领域分组显示所有 action，每个 action 显示其快捷方式和名称
2. **Given** 某个 action 支持 Markdown 模板，**When** 菜单展示该 action，**Then** 该项显示 Markdown 图标标记
3. **Given** 用户点击某个 action（如"创建习惯"），**When** 主显示区加载完成，**Then** 展示基于 manifest `required_fields` 动态生成的表单，包含标题、默认执行时间、默认时长等字段，而非占位文本
4. **Given** 系统已注册 4 个领域（习惯/OKR/任务/时间盒），**When** 用户打开成长领域标签页，**Then** 每个领域分组下均显示该领域定义的 action 列表，无空分组
5. **Given** 某个 action 的表单包含 select 类型字段，**When** 用户点击该字段，**Then** 展示 manifest 中定义的选项列表供用户选择
6. **Given** 用户在 action 表单中填写完毕并点击"确认"，**When** 系统校验通过，**Then** 执行对应操作（创建对象），并显示执行结果
7. **Given** 用户点击非创建类 action（如"激活习惯"），**When** 该 action 未定义 `required_fields`，**Then** 主显示区展示确认界面，显示目标对象摘要（名称、当前状态）和确认/取消按钮

### User Story 5 - AI 对话与编辑区协同工作 (Priority: P3)

用户与 AI 对话过程中，当 AI 解析出结构化的创建意图（如"创建习惯"），主显示区自动分裂为左右两部分：左侧保持 AI 对话，右侧显示对应的表单或 Markdown 编辑器。用户可在编辑区填写内容，也可切换表单/Markdown 模式。确认后执行创建，结果反馈到对话中。

**Why this priority**: 这是 AI 助手从"聊天"升级为"协同工作"的核心体验，但依赖 P1-P4 全部就位。

**Independent Test**: 在对话中输入"帮我创建一个每天跑步的习惯"，验证右侧出现习惯创建表单，填写并确认后执行创建。

**Acceptance Scenarios**:

1. **Given** 用户与 AI 对话中，**When** AI 理解到创建习惯的意图，**Then** 主显示区分裂，右侧显示习惯创建表单，左侧对话继续
2. **Given** 右侧编辑区正在显示表单，**When** 用户切换到 Markdown 标签，**Then** 编辑区切换为 Markdown 编辑器
3. **Given** 右侧为表单模式，**When** AI 在对话中建议修改表单字段，**Then** 表单字段自动更新（如 AI 说"建议名称改为'晨跑'"，名称字段自动填充"晨跑"）
4. **Given** 用户在编辑区点击"确认执行"，**When** 系统校验通过，**Then** 分裂视图折叠，AI 在完整宽度对话中报告执行结果
5. **Given** 用户拖拽分割线，**When** 左右区域宽度调整，**Then** 左右区域按拖拽位置重新分配宽度

---

### User Story 6 - Markdown 模板批量创建 (Priority: P3)

用户下载功能对应的 Markdown 模板，离线或用编辑器填写后上传，系统解析 Markdown 内容并生成对应的结构化创建数据。或用户直接在对话中让 AI 生成 Markdown 初稿，然后手动调整后确认。

**Why this priority**: 增强型功能，适合批量场景和高级用户。MVP 阶段仅支持单个 Markdown 对应单个功能领域。

**Independent Test**: 下载时间盒模板，填写 3 个时间盒，上传后验证系统正确解析出 3 个时间盒数据。

**Acceptance Scenarios**:

1. **Given** 用户处于支持 Markdown 模板的功能界面，**When** 点击"下载模板"，**Then** 浏览器下载对应的 .md 模板文件
2. **Given** 用户上传了已编辑的 Markdown 文件，**When** AI 解析完成，**Then** 编辑区显示解析结果，用户可确认或修正
3. **Given** Markdown 文件存在无法解析的内容，**When** 解析部分失败，**Then** 系统高亮标注有问题的区域，允许用户手动修正；若仍无法解决则降级到表单路径

### User Story 7 - 系统配置管理 (Priority: P3)

用户可通过右上角设置按钮进入配置页面，设置 LLM 服务（厂家、地址、密钥、模型）、时区偏好、习惯模板管理。追踪日志开关也纳入配置页面。

**Why this priority**: 配置是独立功能模块，不阻塞核心用户流程，但 LLM 设置直接影响 AI 助手功能。

**Independent Test**: 进入配置页面，修改时区为"Asia/Tokyo"，验证时间盒显示使用新时区。

**Acceptance Scenarios**:

1. **Given** 用户首次使用应用，**When** 进入 LLM 设置，**Then** 可配置 API 服务商、地址、密钥和默认模型
2. **Given** 用户配置 LLM 服务，**When** 查看配置来源，**Then** 所有提供商列表、模型映射从 `.env` 环境变量加载，前端无硬编码提供商常量
3. **Given** 用户修改了时区，**When** 保存设置，**Then** 所有时间显示立即反映新时区
3. **Given** 用户在习惯模板管理界面，**When** 创建/编辑/删除习惯模板，**Then** 模板变更在下次创建习惯时可使用
4. **Given** 用户点击右上角配置按钮，**When** 主显示区切换，**Then** 显示完整的配置页面视图（含 LLM/时区/习惯模板/追踪日志开关），而非切换追踪日志面板
5. **Given** 用户未配置 LLM，**When** 点击 AI 助手标签页中的"前往设置"按钮，**Then** 主显示区切换到配置页面并定位到 LLM 设置区域

### Edge Cases

- 当快捷方式冲突时，系统在初始化阶段报错并拒绝启动，确保不会在运行时产生歧义
- 当用户快速连续切换会话时，每个会话的状态独立保存，不会相互覆盖
- 当 Markdown 解析完全失败时，系统降级到表单路径，保证用户仍能完成操作
- 当 LLM 未配置时，AI助手标签页显示引导提示和设置跳转按钮，对话输入框隐藏，避免用户困惑
- 当 LLM 配置无效（密钥错误、地址不可达）时，对话发送后返回错误提示，引导用户检查设置或使用模板表单模式
- 当用户同时在左侧面板和主显示区操作时，状态切换遵循"自动保存当前视图再切换"原则
- 当用户正在 AI 对话中，点击左侧成长领域菜单的 action 时：系统自动保存当前对话，主显示区切换到 action 表单视图，对话不丢失（可在左侧会话列表恢复）
- 当网络中断时，会话消息暂存本地，恢复后同步
- 当配置页面打开时，用户点击 Home 按钮应返回时间盒视图
- 当成长领域无已注册 action 时，显示空状态提示而非空白页面
- 当 fetchDomainActions 返回空数据时，系统应在控制台输出诊断信息帮助排查 manifest 加载问题
- 当 action 的 `required_fields` 未定义时，系统显示"该功能暂无可用表单"提示，而非空白页面
- 当 `.env` 中缺少 LLM 配置时，系统使用合理的默认值（如空提供商列表），而非崩溃或渲染异常
- 当动态表单字段包含未知类型时，系统降级为文本输入框

## Requirements *(mandatory)*

### Functional Requirements

**Domain 注册与快捷方式**:

- **FR-001**: 系统必须支持功能领域的快捷方式别名（如 `/createHabit`），作为自然语言输入的快速替代路径
- **FR-002**: 系统必须在启动时校验所有快捷方式别名的全局唯一性，冲突时拒绝启动并报告错误
- **FR-003**: 系统必须支持为每个功能 action 定义视图路由，指定主显示区应加载的界面组件
- **FR-004**: 系统必须从运行时配置（而非硬编码常量）加载所有功能领域的信息，新增功能领域时无需修改核心代码

**AI 会话管理**:

- **FR-005**: 系统必须支持创建、查看、继续、归档、删除 AI 对话会话；归档可恢复，删除不可逆
- **FR-006**: 系统必须在会话切换或导航离开时自动保存对话内容
- **FR-007**: 系统必须在继续旧会话时，向 AI 提供会话期间相关对象的状态变更信息
- **FR-008**: 每个会话必须关联到特定用户（多租户隔离）
- **FR-009**: 会话列表必须按日期分组显示，便于用户浏览历史
- **FR-028**: 当用户尚未配置 LLM 服务时，AI助手标签页必须显示引导提示和跳转至设置页的按钮，对话输入框必须隐藏

**UI 布局**:

- **FR-010**: 系统必须提供三栏布局：顶部导航、左侧面板（固定宽度）、主显示区
- **FR-011**: 左侧面板必须包含 Home 按钮（返回时间盒视图）和两个标签页（AI助手、成长领域）；AI 助手标签页内容仅限会话列表和新对话按钮
- **FR-012**: 主显示区必须支持四种视图状态：时间盒安排（默认）、AI 对话、功能 action 界面、配置页面
- **FR-013**: 当 AI 解析出结构化创建意图时，主显示区必须能分裂为对话区 + 编辑区两部分
- **FR-014**: 分裂视图的分割线必须可拖拽调整宽度，左侧对话区保持最小宽度约束

**成长领域菜单**:

- **FR-015**: 成长领域菜单必须从功能注册表动态生成，按领域分组显示所有可用的 action
- **FR-016**: 每个菜单项必须显示 action 名称和快捷方式
- **FR-017**: 点击菜单项后，主显示区必须加载对应的功能界面
- **FR-039**: 每个域 action 点击后，主显示区必须根据 manifest 的 `required_fields` 定义动态生成对应的表单界面，而非显示占位文本
- **FR-040**: 支持表单字段类型必须覆盖 manifest `required_fields` 中定义的所有类型（text、number、time、date、select 等）
- **FR-041**: 当域 action 配置了 `templates.markdown` 时，表单界面必须同时提供 Markdown 模板上传入口作为替代填写方式
- **FR-042**: 非创建类 action（如激活、暂停、结束等状态变更操作）若未定义 `required_fields`，点击后主显示区必须展示确认界面，包含目标对象的摘要信息和确认/取消按钮
- **FR-043**: 动态表单和确认界面的提交必须将用户输入映射为 StructuredIntent，通过现有 Intent Engine（onValidate → 状态机）统一执行，不引入新的提交路径

**Markdown 工作流**:

- **FR-018**: 支持 Markdown 模板的功能必须提供模板下载按钮
- **FR-019**: 系统必须支持上传 .md 文件，由 AI 解析为结构化数据
- **FR-020**: Markdown 解析失败时，系统必须高亮问题区域并提供降级到表单填写的路径
- **FR-021**: MVP 阶段每个 Markdown 文件仅产生单个功能领域的单个操作

**配置管理**:

- **FR-022**: 系统必须提供 LLM 服务配置界面（服务商、地址、密钥、模型）
- **FR-023**: 系统必须提供时区选择器，默认从浏览器自动检测
- **FR-024**: 系统必须保护 API 密钥的存储安全（客户端加密，不通过服务端中转）
- **FR-025**: 配置页面必须集成习惯模板管理功能
- **FR-037**: LLM 配置必须统一存储在 `.env` 文件中（提供商列表、模型映射、API 地址），消除前端代码中的硬编码常量；前端通过标准环境变量读取机制获取配置值
- **FR-038**: 新增或变更 LLM 提供商时，仅需修改 `.env` 文件和对应的 Next.js 环境变量前缀，无需修改 `config.ts` 的代码逻辑

**配置页面**:

- **FR-029**: 配置页面必须作为主显示区的独立视图类型（与时间盒视图、对话视图并列），包含：通用设置（占位）、LLM 设置、时区设置、习惯模板管理、追踪日志开关
- **FR-030**: 右上角配置按钮必须导航到配置页面视图，而非切换追踪日志面板
- **FR-031**: 追踪日志开关必须从 TopNav 按钮迁移到配置页面的通用设置区域内

**回溯修正**:

- **FR-032**: 左侧面板 assistant 标签必须仅包含会话列表（SessionList）和新对话按钮，禁止包含任何输入框、模式切换（AI对话/表单填写）或确认弹窗
- **FR-033**: 主显示区 schedule 视图必须仅展示时间盒日历（DateNav + DayView/WeekView/MonthView），禁止展示习惯库/模板/OKR/项目等标签页导航
- **FR-034**: 习惯库、模板、OKR、项目/任务等视图必须仅通过成长领域菜单的 action 进入（对应 `mainViewState.type = 'action'`），不再作为 schedule 的子标签页
- **FR-035**: `fetchDomainActions()` 必须正确返回所有已注册领域的 action 列表；数据来源为 Registry 运行时从 manifest 加载，不允许硬编码
- **FR-036**: LLM 未配置提示中的"前往设置"按钮必须成功导航到配置页面，且定位到 LLM 设置区域

**文件处理**:

- **FR-026**: 系统必须支持在对话中上传 `.md`、`.txt`、`.csv`、`.xlsx`、`.xls` 文件
- **FR-027**: 上传的文件内容必须注入到 AI 对话上下文中

### Key Entities

- **AI 会话 (AI Session)**: 代表用户与 AI 的一次完整对话，包含消息列表、会话状态（活跃→已归档→已删除）、引用的对象ID、状态快照。归档可恢复至活跃状态，删除不可逆。属于基础设施对象，非业务对象。
- **会话消息 (Chat Message)**: 会话中的单条消息，包含角色（用户/助手/系统）、内容、时间戳、关联的意图ID。
- **功能清单 (Domain Manifest)**: 定义每个功能领域的行为、快捷方式、视图路由、模板的声明式配置。
- **用户设置 (User Settings)**: 存储用户的 LLM 配置、时区偏好、界面偏好等个人设置。
- **视图路由 (View Route)**: 定义功能 action 到界面组件的映射关系。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户通过快捷方式触达目标功能界面在 1 秒内完成
- **SC-002**: 会话切换后消息和上下文恢复完整率 100%（不丢失消息）
- **SC-003**: 新增一个功能领域并注册到菜单的操作不超过更新一个声明式配置文件（无需修改核心代码）
- **SC-004**: 用户从输入意想到看到编辑区（分裂视图）的完整流程在 3 秒内完成
- **SC-005**: Markdown 模板下载→离线编辑→上传→解析→确认执行的完整闭环可被单用户独立完成
- **SC-006**: 快捷方式冲突在系统启动时即被检测并阻止，运行时零冲突
- **SC-007**: LLM 配置错误时系统不崩溃，降级到模板表单模式继续提供服务
- **SC-008**: 所有 4 个域（习惯/OKR/任务/时间盒）的主要创建类 action 点击后均能加载对应的动态表单，不再显示占位文本
- **SC-009**: 新增 LLM 提供商仅需修改 `.env` 文件，无需修改前端代码中的常量定义

## Assumptions

- 目标用户为 Web 桌面端用户，移动端适配不在本次范围
- 用户具有稳定的网络连接，文件上传大小限制在 10MB 以内
- 现有的数据库和对象模型基础设施可复用
- LLM 提供商配置为服务端环境变量（`.env`），通过 Next.js API 路由暴露给前端，前端不直接读取 `.env` 文件
- 动态表单基于 manifest 的 `required_fields` 生成，字段类型映射到标准 UI 组件（text→Input, number→Input[type=number], time→TimePicker, date→DatePicker, select→Select）
- LLM API 密钥由用户自行获取和管理，系统不提供内置密钥
- 跨功能领域的 Markdown 批处理（一个 Markdown 文件同时创建习惯和任务）不在 MVP 范围内
- 多语言国际化为预留框架，本次仅中文界面
- 桥接层（Bridge Layer）为 Phase 2 工作，本次不实现
- AI 对话的实时流式响应为后续优化项，MVP 可接受非流式响应
- 动态表单提交复用现有 StructuredIntent 机制，表单字段名与 intent 参数名保持一致，由前端映射层负责转换
