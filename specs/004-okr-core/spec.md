# Feature Specification: OKR 核心管理 (004-okr-core)

**Feature Branch**: `004-okr-core`
**Created**: 2026-05-10
**Updated**: 2026-05-11
**Status**: Draft
**Input**: OKR 管理功能核心阶段 — 实现 OKR 的 CRUD、生命周期管理、KR 进度推进、双栏联动界面、目标编号与优先级体系，为后续 AI 智能功能（004b）提供数据基础

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 创建和编辑 OKR (Priority: P1)

用户希望规划本季度目标。打开 OKR 页面，点击"创建 Objective"，填写标题、描述、选择类型（愿景型/承诺型）、重要程度（P0 必须完成 / P1 应该完成 / P2 有余力则做）、选择周期层次（年度/半年度/季度/月度）。系统根据周期层次自动填充起止日期，并自动生成目标编号（如 26Q1-O1）。用户可微调日期后保存为草稿。随后进入 OKR 编辑面板，逐条添加 Key Result（目标值 + 单位），完成 OKR 框架的搭建。编辑过程中可随时修改各字段内容。

**Why this priority**: OKR 的创建是最基础的入口，没有 OKR 数据就无法进行任何后续操作。这是整个功能的价值起点。

**Independent Test**: 可通过创建一个包含 3 个 KR 的 Objective 并验证所有字段（含编号、优先级、周期日期自动填充）正确保存来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户在 OKR 工作区，**When** 点击创建并填写标题、类型、优先级、周期层次后保存，**Then** 系统创建一个 draft 状态的 Objective，自动分配编号（如 26Q1-O1），周期日期自动填充，工作区左栏显示该条目
2. **Given** 用户创建 OKR 时选择周期层次为"季度"，**When** 当前为 2026 年 5 月，**Then** 系统自动填充起止日期为 2026-04-01 ~ 2026-06-30
3. **Given** 用户创建 OKR 时选择周期层次为"半年度"，**When** 当前为 2026 年 5 月，**Then** 系统自动填充起止日期为 2026-01-01 ~ 2026-06-30（上半年）
4. **Given** 用户创建 OKR 时选择周期层次为"月度"，**When** 当前为 2026 年 5 月，**Then** 系统自动填充起止日期为 2026-05-01 ~ 2026-05-31
5. **Given** 一个 draft 状态的 Objective，**When** 用户添加 Key Result（标题、目标值、单位），**Then** KR 以 draft 状态关联到该 Objective，右栏详情显示新增的 KR
6. **Given** 一个 draft 状态的 Objective，**When** 用户编辑标题或修改周期范围，**Then** 修改即时保存并反映在详情面板
7. **Given** 一个 draft 状态的 KR，**When** 用户删除该 KR（经二次确认），**Then** KR 被移除，Objective 的 KR 列表更新
8. **Given** 用户在同一周期层次下创建第二个 Objective，**When** 保存后，**Then** 编号自增为 O2（如 26Q1-O2）

---

### User Story 2 - 激活和管理 OKR 生命周期 (Priority: P1)

用户完成 OKR 框架搭建后，点击"激活"将 OKR 投入执行。系统校验激活条件（至少 1 个 KR、日期合理、标题非空），通过后 Objective 和所有 draft KR 变为 active 状态。执行期间用户可暂停（暂时搁置）、恢复（重新激活）、完成（标记达成）、废弃（放弃）OKR。最终可将已结束的 OKR 归档以保持列表整洁。所有删除和废弃操作均需二次确认。

**Why this priority**: 生命周期管理是 OKR 功能的核心差异化能力，没有状态流转，OKR 只是静态文本。与创建同等重要，但依赖创建先完成。

**Independent Test**: 可通过创建 → 激活 → 暂停 → 恢复 → 完成 → 归档的完整链路来独立测试状态机。

**Acceptance Scenarios**:

1. **Given** 一个 draft Objective（含 1+ KR、日期已设置），**When** 用户点击激活，**Then** Objective 和所有 draft KR 变为 active 状态，变更历史记录 ObjectiveActivated 事件
2. **Given** 一个 draft Objective（无 KR），**When** 用户点击激活，**Then** 系统拒绝并提示"至少需要 1 个 Key Result"
3. **Given** 一个 active Objective，**When** 用户点击暂停，**Then** Objective 和所有 active KR 变为 paused 状态
4. **Given** 一个 paused Objective，**When** 用户点击恢复，**Then** Objective 和所有 paused KR 变为 active 状态
5. **Given** 一个 active Objective，**When** 用户点击完成，**Then** Objective 和所有 KR 变为 completed 状态
6. **Given** 一个 active 或 paused Objective，**When** 用户点击废弃，**Then** 弹出确认对话框，确认后 Objective 和所有 KR 变为 discarded 状态
7. **Given** 一个 completed 或 discarded Objective，**When** 用户点击归档，**Then** Objective 和所有 KR 变为 archived 状态，不再出现在默认列表视图中

---

### User Story 3 - 推进和查看 KR 进度 (Priority: P2)

用户在 OKR 详情面板查看各 KR 的进度条和当前值/目标值。可手动输入当前进度值更新 KR。当 KR 关联了任务或习惯时，任务完成或习惯打卡会自动推进 KR 进度。当 KR 的当前值达到目标值时，KR 自动标记为 completed。当所有 KR 都完成时，系统提示用户可以将 Objective 标记为完成。

**Why this priority**: 进度追踪是 OKR 执行阶段的核心价值，但依赖 OKR 先被激活（P1）。手动更新是基础能力，自动推进是增值能力。

**Independent Test**: 可通过激活一个 OKR 后手动更新 KR 进度到 100%，验证 KR 自动完成来独立测试。

**Acceptance Scenarios**:

1. **Given** 一个 active KR（目标值 100），**When** 用户手动输入当前值 60，**Then** 进度条更新为 60%，KR 状态保持 active
2. **Given** 一个 active KR（目标值 100），**When** 用户手动输入当前值 100，**Then** 进度条显示 100%，KR 自动标记为 completed
3. **Given** 一个 active KR 关联了 3 个任务（目标值 30），**When** 用户完成其中 1 个任务，**Then** KR 当前值增加 10（等分策略），进度更新为 33%
4. **Given** 一个 active KR 关联了一个习惯（目标值 5），**When** 用户打卡该习惯 1 次，**Then** KR 当前值增加 1
5. **Given** 一个 Objective 的所有 KR 都已 completed，**When** 最后一个 KR 自动完成时，**Then** 系统发出提示建议将 Objective 标记为完成

---

### User Story 4 - 双栏联动的 OKR 工作区 (Priority: P2)

用户进入 OKR 页面，看到左右双栏布局。左栏为 OKR 目录，顶部有状态标签页（全部/草稿/已暂停/已完成/已废弃），下方按周期层次分组显示 OKR 条目。每个条目显示编号 + 标题 + 状态标签 + 重要程度。点击左栏条目，右栏展示该 OKR 的详情信息（标题、元数据、KR 卡片列表）。右栏支持四种模式：空状态引导、详情视图、编辑表单、新增表单。用户可在左栏直接执行操作（激活/编辑/删除/归档等），编辑操作会切换右栏到编辑模式并预填数据（含已有 KR）。

**Why this priority**: 双栏布局是 OKR 日常使用的高频交互方式，提供一览-操作一体的流畅体验。优先级稍低于创建和生命周期管理，因为用户首先需要有 OKR 数据。

**Independent Test**: 可通过创建多个不同状态的 OKR 后验证双栏联动（左栏选择、右栏展示、操作切换）来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户有多个不同状态的 OKR，**When** 进入 OKR 页面，**Then** 左栏按状态标签页分组显示，右栏显示空状态引导
2. **Given** 左栏有多条 OKR，**When** 点击某条 OKR，**Then** 右栏展示该 OKR 的详情（标题、元数据、KR 卡片列表）
3. **Given** 左栏某条 OKR，**When** 点击编辑按钮，**Then** 右栏切换到编辑模式，表单预填当前数据（含已有 KR 信息）
4. **Given** 左栏某条 OKR，**When** 点击删除按钮，**Then** 弹出确认对话框，确认后 OKR 标记为已废弃
5. **Given** 左栏状态标签页切换到"草稿"，**When** 操作完成，**Then** 仅显示草稿状态的 OKR
6. **Given** "全部"标签页，**When** 进入 OKR 页面，**Then** 显示所有非归档状态的 OKR（包含草稿、进行中、已暂停、已完成、已废弃）
7. **Given** 用户在右栏编辑模式保存了修改，**When** 保存完成，**Then** 左栏列表通过局部更新即时反映变化，不出现空白闪烁
8. **Given** 左栏有多条 OKR，**When** 查看列表，**Then** 条目按周期层次分组（如季度组、年度组），每组内显示编号 + 标题 + 状态 + 重要程度
9. **Given** 用户在 OKR 详情面板中，**When** 添加一个新 KR，**Then** 新 KR 立即出现在 KR 列表中，无需刷新页面
10. **Given** 用户在 OKR 详情面板中，**When** 删除一个 draft KR，**Then** 该 KR 立即从列表中消失，无需刷新页面
11. **Given** 用户进入 OKR 工作区，**When** 查看右栏面板，**Then** 右栏占满主内容区域全部可用宽度，与系统整体布局风格统一

---

### User Story 5 - 关联任务和习惯到 KR (Priority: P3)

用户在 KR 详情中点击"关联任务"或"关联习惯"，选择已有项目进行关联。关联后，该任务/习惯的完成或打卡会自动推进 KR 进度。用户也可以取消关联。在任务/习惯详情页可以看到所属的 OKR 信息。

**Why this priority**: 关联能力是自动进度推进的前提，但用户可以通过手动更新进度来替代，因此优先级低于手动进度管理。

**Independent Test**: 可通过创建任务后关联到 KR，完成任务验证进度自动推进来独立测试。

**Acceptance Scenarios**:

1. **Given** 一个 active KR 和一个未完成的任务，**When** 用户将任务关联到该 KR，**Then** KR 详情中显示关联的任务，任务详情页显示所属 OKR
2. **Given** 一个 KR 关联了一个任务，**When** 用户取消关联，**Then** KR 详情中不再显示该任务，后续任务完成不再推进该 KR
3. **Given** 一个任务已关联到某个 KR，**When** 在任务详情页查看，**Then** 显示所属 Objective 标题和 KR 标题

---

### Edge Cases

- KR 的目标值为 0 或负数时如何处理？系统应拒绝并提示"目标值必须大于 0"
- 手动输入的当前值超过目标值时如何处理？系统应限制在 [0, targetValue] 范围内
- Objective 周期的结束日期早于开始日期时如何处理？系统应拒绝并提示日期不合理
- 关联任务被删除时，KR 进度是否需要重新计算？需要，系统应重新计算等分
- 所有 KR 被删除后，active 状态的 Objective 如何处理？保持 active 但提示需要添加 KR
- 已归档的 OKR 是否可以取消归档？MVP 阶段不支持，归档为终态
- KR 关联的任务数量为 0 时，手动更新是否仍然可用？是，手动更新始终可用
- 用户尝试对已归档 OKR 执行操作时如何响应？系统应提示该 OKR 已归档，不可操作
- 同一周期层次下删除中间编号的 OKR 后编号是否重排？不重排，编号一旦分配保持不变，新 OKR 按当前最大序号自增
- 用户手动修改了自动填充的起止日期后切换周期层次，**When** 如何处理？系统应以新选择的周期层次重新计算覆盖，用户需重新微调
- 编辑模式下切换到另一条 OKR 时未保存的修改如何处理？系统应提示有未保存的修改，由用户决定是否放弃
- KR 添加/删除操作后数据已写入数据库但 UI 未刷新时如何处理？系统应在操作成功回调中即时更新本地状态，不依赖全量刷新

## Requirements *(mandatory)*

### Functional Requirements

**创建与编辑**

- **FR-001**: 系统必须允许用户创建 Objective，包含标题、描述、类型（愿景型/承诺型）、重要程度（P0/P1/P2）、周期层次（年度/半年度/季度/月度）、起止日期
- **FR-002**: 系统必须允许用户为 Objective 添加 Key Result，包含标题、目标值（数值）、单位
- **FR-003**: 系统在创建 Objective 时必须自动生成目标编号，格式为 {年度缩写}{周期代码}-O{序号}（如 26Q1-O1、26H2-O1、26M05-O1、26Y-O1）
- **FR-004**: 系统在创建 Objective 时必须默认设置重要程度为 P1
- **FR-005**: 系统在选择周期层次后必须自动填充对应的起止日期（年度：1/1~12/31；半年度：1/1~6/30 或 7/1~12/31 按当前月份判断；季度：当前季度起止月；月度：当前月份起止日）
- **FR-006**: KR 编号采用展示级拼接格式（如 O1-K1, O1-K2），不单独存储，按同 Objective 下 KR 顺序生成

**生命周期**

- **FR-007**: 系统必须支持 Objective 的完整生命周期：draft → active → paused → completed/discarded → archived
- **FR-008**: 系统在激活 Objective 前必须校验：至少 1 个 KR、周期日期已设置且合理（end > start）、标题非空
- **FR-009**: 系统在 Objective 状态变更时必须同步更新所有关联 KR 的状态
- **FR-010**: 系统必须记录所有状态变更事件（创建、激活、暂停、恢复、完成、废弃、归档）
- **FR-011**: 系统必须在所有删除和废弃操作（Objective 和 KR）时弹出二次确认对话框，对话框需明确说明操作后果

**KR 进度**

- **FR-012**: 系统必须允许用户手动更新 KR 的当前进度值
- **FR-013**: 系统在 KR 当前值达到目标值时必须自动将 KR 标记为 completed
- **FR-014**: 系统必须在任务完成时自动推进关联 KR 的进度（等分策略：每个关联任务贡献 targetValue / 关联任务总数）
- **FR-015**: 系统必须在习惯打卡时自动推进关联 KR 的进度（每次打卡 currentValue +1）
- **FR-016**: 系统必须在所有 KR 完成时提示用户可将 Objective 标记为完成
- **FR-017**: 系统必须限制 KR 进度值在 [0, targetValue] 范围内
- **FR-018**: 系统必须仅允许在 draft 状态下删除 KR

**界面与交互**

- **FR-019**: 系统必须采用双栏联动布局：左栏为 OKR 目录（固定宽度），右栏为操作面板（弹性宽度）
- **FR-020**: 系统左栏必须支持按状态筛选（全部/草稿/已暂停/已完成/已废弃），"全部"标签页显示所有非归档状态的 OKR
- **FR-021**: 系统左栏必须按周期层次分组显示 OKR 条目
- **FR-022**: 系统必须在左栏每个 OKR 条目上显示编号、标题、状态标签和重要程度
- **FR-023**: 系统右栏必须支持四种模式：空状态引导、详情视图、编辑表单（预填数据含已有 KR）、新增表单
- **FR-024**: 系统编辑保存后必须通过局部状态更新刷新列表，不重新全量获取数据
- **FR-025**: 系统必须在 OKR 详情页显示变更历史记录
- **FR-026**: 系统必须允许用户关联/取消关联任务和习惯到 KR
- **FR-027**: 归档的 OKR 在默认列表视图中隐藏，需通过筛选条件查看

**数据模型**

- **FR-028**: 系统必须为 Objective 和 KR 新增 discarded 状态，并在废弃时记录时间戳
- **FR-029**: 系统必须支持半年度（semi_annual）周期层次，OKR 表单仅展示年度/半年度/季度/月度四个选项（daily/weekly 保留于枚举中供其他领域使用）
- **FR-030**: 编号中的序号按同一前缀下已有 Objective 数量自增，编号一旦分配不再变更

**KR 即时刷新**

- **FR-031**: 系统在用户添加 KR 后必须即时在详情面板中显示新 KR，无需刷新页面或手动重新加载
- **FR-032**: 系统在用户删除 KR 后必须即时从详情面板中移除该 KR，无需刷新页面或手动重新加载

**布局优化**

- **FR-033**: OKR 工作区右栏面板必须占满主内容区域的全部可用宽度，与系统整体布局风格统一

### Key Entities

- **Objective（目标）**: 用户设定的周期性目标，包含标题、描述、类型（愿景型/承诺型）、重要程度（P0/P1/P2）、目标编号（自动生成，用户级唯一）、状态、周期范围（含周期类型）。可包含多个 Key Result。支持层级关系（父级 Objective）。生命周期状态：draft / active / paused / completed / discarded / archived
- **Key Result（关键结果）**: 量化 Objective 达成程度的指标，包含标题、目标值、当前值、单位、进度百分比。归属于一个 Objective。可关联任务和习惯以自动推进进度。展示级编号（如 O1-K1）不单独存储。生命周期与所属 Objective 同步
- **OKR 事件**: 记录 Objective 和 KR 的所有状态变更和进度更新事件，用于变更历史展示和后续分析

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户能在 2 分钟内完成一个包含 3 个 KR 的 Objective 创建并保存为草稿
- **SC-002**: Objective 的所有生命周期状态转换操作在 1 秒内完成响应
- **SC-003**: KR 进度更新（手动和自动）在操作完成后 2 秒内反映到 UI 上
- **SC-004**: 关联任务完成或习惯打卡后，KR 进度自动推进正确率 100%（等分计算无误差）
- **SC-005**: 用户能在 OKR 工作区中通过一次点击查看任何 OKR 的详情，无需页面跳转
- **SC-006**: 编辑保存后 OKR 列表在 1 秒内完成局部更新，不出现空白闪烁
- **SC-007**: 目标编号生成准确率 100%，无重复编号
- **SC-008**: 90% 的 OKR 生命周期操作（激活、暂停、完成等）能由用户在首次尝试时成功完成
- **SC-009**: "全部"筛选正确显示所有非归档状态的 OKR，不遗漏任何状态类型
- **SC-010**: KR 添加或删除操作后，详情面板在 1 秒内即时反映变化，无需用户手动刷新
- **SC-011**: OKR 工作区右栏面板充分利用主内容区域全部宽度，与系统其他视图（如时间安排）布局风格一致

## Assumptions

- MVP 阶段仅支持 Web 端，移动端适配在后续迭代中完成
- OKR 类型（愿景型/承诺型）由用户手动选择，不做 AI 自动判定
- KR 进度自动推进采用等分策略（每个关联任务等权重），不支持自定义权重
- 废弃状态的数据不物理删除，归档后从默认视图中隐藏但数据保留
- MVP 不支持取消归档操作，归档为终态
- 所有数据操作遵循多租户隔离，通过 userId 确保数据安全
- 文件导入和 AI 质量评估推迟到 004b-okr-intelligence 切片
- 任务和习惯的 keyResultId 外键已在现有 schema 中存在，无需新增
- 事件记录复用现有 system_events 表，不新建表
