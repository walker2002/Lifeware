# Feature Specification: 时间盒纵向薄切片（Timebox Vertical Slice）

**Feature Branch**: `002-timebox-slice`
**Created**: 2026-05-03
**Status**: Draft
**Input**: 用户描述：构建时间盒纵向薄切片，实现"我今天10:00开始做市场调研报告，花费2小时"的完整端到端链路，包括意图解析、规则校验、状态机、事件总线、行动切面、编排器和时间盒 Domain 插件。

## User Scenarios & Testing

### User Story 1 - 自然语言创建时间盒 (Priority: P1)

用户打开系统首页，在输入框中输入"我今天10:00开始做市场调研报告，花费2小时"，系统识别意图后创建一个计划状态的时间盒，并在页面上展示创建结果和时间盒详情。

**Why this priority**: 这是系统的核心价值链路——从自然语言到结构化时间安排。它能端到端验证 Intent Engine、Rule Engine、State Machine 的协同工作，是整个 Nexus 架构的第一条贯通管道。

**Independent Test**: 可通过输入自然语言、观察页面是否展示正确的时间盒卡片来独立验证，不依赖其他功能。

**Acceptance Scenarios**:

1. **Given** 用户在首页输入框中输入"我今天10:00开始做市场调研报告，花费2小时"，**When** 提交意图，**Then** 系统解析为结构化意图（标题="市场调研报告"，开始时间=今天10:00，时长=120分钟），创建 `status: planned` 的时间盒，并在页面上展示时间盒卡片。
2. **Given** 用户输入的意图缺少关键信息（如"我要做报告"没有时间），**When** 系统尝试解析，**Then** 返回解析失败提示，引导用户使用表单模式补充信息。
3. **Given** 用户输入的意图解析置信度低于阈值，**When** 系统返回解析结果，**Then** 页面显示解析预览并请求用户确认后再创建。

---

### User Story 2 - 表单模式创建时间盒 (Priority: P1)

用户不使用自然语言，而是通过表单直接填写标题、开始时间、持续时长来创建时间盒。这是 AI 解析失败时的保底路径。

**Why this priority**: Constitution VIII 要求 AI 失败时必须有 template-form fallback。这是系统的可靠性保障，也是独立于 AI 的纯功能路径。

**Independent Test**: 可通过表单填写并提交、观察时间盒是否正确创建来独立验证。

**Acceptance Scenarios**:

1. **Given** 用户在表单中填写标题"产品方案设计"、开始时间"明天09:00"、持续时长"120分钟"，**When** 提交表单，**Then** 系统直接创建 `status: planned` 的时间盒，不经过 AI 解析。
2. **Given** 用户在表单中未填写标题，**When** 提交表单，**Then** 系统提示"标题不能为空"并阻止提交。

---

### User Story 3 - 时间重叠规则校验 (Priority: P2)

当用户尝试创建的时间盒与已有时间盒时间重叠时，系统提示警告并等待用户确认。

**Why this priority**: 规则校验是系统能量优先调度理念的基础。时间不可重叠是最基本的规则，验证 Rule Engine 的"教练而非守门员"模式（警告而非阻止）。

**Independent Test**: 先创建一个时间盒，再尝试创建一个时间重叠的时间盒，观察是否出现警告提示。

**Acceptance Scenarios**:

1. **Given** 已存在一个10:00-12:00的时间盒，**When** 用户尝试创建11:00-13:00的时间盒，**Then** 系统返回"时间重叠"警告，显示冲突信息，等待用户确认后再创建。
2. **Given** 已存在一个10:00-12:00的时间盒，**When** 用户尝试创建12:00-14:00的时间盒（紧接不重叠），**Then** 系统直接通过，无警告。

---

### User Story 4 - 行动切面展示 (Priority: P2)

时间盒创建成功后，系统在 Dynamic Tile 区域展示成功提示。当时间盒即将开始（距开始时间小于15分钟）时，系统展示提醒。

**Why this priority**: Action Surface 是系统与用户持续交互的核心机制，验证系统能主动提供有价值的行动提示。

**Independent Test**: 创建时间盒后观察页面 Dynamic Tile 区域是否有提示；调整系统时间距开始时间<15分钟后观察是否有开始提醒。

**Acceptance Scenarios**:

1. **Given** 时间盒创建成功，**When** 页面刷新，**Then** Dynamic Tile 区域显示"时间盒已创建: 市场调研报告"。
2. **Given** 存在一个 planned 状态的时间盒且距开始时间<15分钟，**When** 页面加载，**Then** 行动提示区显示"即将开始: 市场调研报告"。

---

### User Story 5 - 应用界面框架搭建 (Priority: P0)

这是系统的第一个界面。在实现具体功能之前，必须先建立应用的整体布局框架，包含导航结构、内容区域的分区规范，以及基础的设计令牌（颜色、字体、间距）。框架搭好后，后续功能（输入框、时间盒卡片、行动切面）直接填充到对应区域中。

**Why this priority**: 这是所有 UI 的基础。没有布局框架，User Story 1-4 的界面无处安放。P0 意味着必须最先完成。

**Independent Test**: 可通过打开浏览器验证布局结构是否完整：顶部导航栏存在、左右两栏内容区域可见、各区域有明确的标题标识。

**Acceptance Scenarios**:

1. **Given** 用户打开应用首页，**When** 页面加载完成，**Then** 页面呈现 Notion 风格布局：顶部固定导航栏 → 行动切面横幅（条件渲染） → 左侧 AI 交互面板 + 右侧主内容区（默认日视图三栏，顶部日期导航支持日/周/月切换）。
2. **Given** 页面加载完成，**When** 用户查看页面顶部，**Then** 显示应用名称和极简图标导航栏。
3. **Given** 页面在不同屏幕宽度下，**When** 窗口缩小至移动端宽度，**Then** 布局响应式折叠为单栏，AI 面板折叠为可展开面板。

---

### User Story 6 - 三栏时间盒视图 (Priority: P1)

用户进入系统后直接看到当天的时间盒。主内容区采用统一的三栏视图取代之前的"今日模式/日历模式"切换。顶部提供日期导航，支持"日/周/月"三种浏览模式，默认显示"日"视图。

**Why this priority**: 这是时间盒管理的核心交互界面。三栏视图让用户同时看到列表、时间轴和日历，信息密度高且无需切换模式。

**Independent Test**: 打开系统首页，验证默认显示当天日视图（左列表 + 中间时间轴 + 右日历），切换到周/月模式后显示对应日历，前后翻页可切换日期。

**Acceptance Scenarios**:

1. **Given** 用户打开系统首页，**When** 页面加载完成，**Then** 主内容区默认显示"日"视图，呈现三栏布局：左侧为当天时间盒列表、中间为小时时间轴（时间盒以色块显示）、右侧为月历小日历。
2. **Given** 日视图已显示，**When** 用户点击顶部"周"按钮，**Then** 视图切换为周模式，显示当周的日历时间表格，每列代表一天，时间盒显示在对应时段，数据同步更新为当周范围。
3. **Given** 日视图已显示，**When** 用户点击顶部"月"按钮，**Then** 视图切换为月模式，显示当月日历网格，时间盒以事件块形式显示在对应日期，数据同步更新为当月范围。
4. **Given** 日视图已显示，**When** 用户点击前进/后退按钮，**Then** 日期导航切换到前一天/后一天，时间盒列表和时间轴同步更新。
5. **Given** 周视图已显示，**When** 用户点击前进/后退按钮，**Then** 日历切换到前一周/后一周。
6. **Given** 月视图已显示，**When** 用户点击前进/后退按钮，**Then** 日历切换到前一月/后一月。
7. **Given** 用户在移动端宽度下查看页面，**When** 页面加载完成，**Then** 顶部导航仅显示"日/月"两个选项（隐藏"周"），日视图折叠为单栏。
8. **Given** 日视图已显示，**When** 用户点击右侧月历小日历中的某一天，**Then** 日视图跳转到该日期，时间盒列表和时间轴同步更新为该日数据。

---

### Edge Cases

- 用户输入的时间是过去时间时，系统如何处理？→ 返回错误"开始时间必须在当前时间之后"。
- 用户输入的持续时长为0或极大值（如48小时）时，系统如何处理？→ Rule Engine 返回时长不合法警告。
- AI 解析服务不可用时，系统如何处理？→ 自动切换到表单模式，提示用户"AI 服务暂时不可用，请使用表单创建"。
- 同一时间精确重叠（两个时间盒都是10:00-12:00）时，系统如何处理？→ 返回时间重叠警告。

## Requirements

### Functional Requirements

- **FR-001**: 系统必须提供自然语言输入框，接受用户的自由文本时间盒意图描述。
- **FR-002**: 系统必须将自然语言输入解析为结构化意图，包含标题、开始时间、持续时长三个核心字段。
- **FR-003**: 系统必须在 AI 解析失败或不可用时，提供表单模式作为替代输入方式。
- **FR-004**: 系统必须对结构化意图执行规则校验，至少包含时间重叠检测和时长合法性校验。
- **FR-005**: 规则校验结果必须是"通过/警告/需确认"三种之一，不得直接阻止用户操作。
- **FR-006**: 系统必须通过状态机管理时间盒生命周期，初始状态为 `planned`，支持 planned→running→paused→ended→logged 的转移路径。
- **FR-007**: 系统必须在状态变更后发布系统事件（如 TimeboxCreated）。
- **FR-008**: 系统必须基于当前状态和事件生成行动切面（Dynamic Tile 和 Continuity Cue）。
- **FR-009**: 编排器必须按固定顺序调度各组件：Intent Engine → Rule Engine → State Machine → Event Bus → Action Surface Engine。
- **FR-010**: 时间盒 Domain 插件必须实现 `onValidate`（结构性验证）和 `onEvent`（事件响应）两个钩子。
- **FR-011**: 所有写入操作必须通过意图引擎入口，其他组件不得直接创建或修改时间盒（Constitution I）。
- **FR-012**: 系统必须在首页同时展示：输入区域、行动切面区域、已有时间盒列表。
- **FR-013**: 系统必须建立 Notion 风格整体界面布局框架：顶部导航栏 + 行动切面横幅（TilesBanner） + 左侧 AI 交互面板 + 右侧主内容区，作为后续所有功能页面的基础骨架。主内容区采用统一的三栏时间盒视图，不再使用"今日模式/日历模式"切换。
- **FR-014**: 界面必须使用设计令牌（Design Tokens）定义颜色、字体、间距等基础样式变量，确保视觉一致性。设计令牌参照 `DESIGN.md` 中的暖色奶油画布 + 珊瑚色主调体系。
- **FR-015**: 布局必须支持响应式适配，在移动端宽度下两栏折叠为单栏，AI 面板变为可展开面板。
- **FR-016**: 主内容区顶部必须提供日期导航栏，包含日期模式切换（日/周/月，默认"日"）和前进/后退翻页按钮，用于切换浏览的日期范围。
- **FR-017**: 日视图必须采用三栏布局：左侧显示选中日期的时间盒列表，中间显示小时时间轴（时间盒以状态色块形式定位在对应时段），右侧显示月历小日历。周视图显示当周的日历时间表格，月视图显示当月日历网格。
- **FR-018**: 进入系统后主内容区必须默认显示当天日期的时间盒数据，日期导航默认选中"日"模式。
- **FR-019**: 移动端宽度下，日期导航仅显示"日/月"两个选项（隐藏"周"），日视图三栏折叠为单栏布局。

### Key Entities

- **Intention（意图）**: 用户原始输入的记录，包含原始文本、输入模式、捕获时间、状态（captured→dissolved）。
- **StructuredIntent（结构化意图）**: 意图引擎解析后的结构化结果，包含目标域（timebox）、动作（create_timebox）、字段（标题/开始时间/时长）、置信度。
- **Timebox（时间盒）**: 核心业务实体，包含标题、开始/结束时间、状态（planned/running/paused/ended/logged）、关联的任务和习惯。
- **SystemEvent（系统事件）**: 状态变更后发布的事件，如 TimeboxCreated、TimeboxStarted 等。
- **ActionSurface（行动切面）**: 系统基于当前状态生成的行动建议，分为 Guide（持久指引）、Tile（动态磁贴）、Cue（连续性提示）三类。
- **ContextSnapshot（上下文快照）**: 状态机执行后生成的系统状态快照，供行动切面引擎和 Domain 插件只读使用。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户输入"我今天10:00开始做市场调研报告，花费2小时"后，5秒内页面展示创建成功的时间盒卡片。
- **SC-002**: AI 解析不可用时，用户通过表单模式在30秒内完成时间盒创建。
- **SC-003**: 时间重叠的场景中，系统在2秒内返回警告信息，用户可确认后继续创建。
- **SC-004**: 时间盒创建后，Dynamic Tile 区域立即展示对应的行动提示。
- **SC-005**: 端到端链路无遗漏：用户输入 → 意图解析 → 规则校验 → 状态创建 → 事件发布 → 行动展示，每个环节都有可观测的输出。
- **SC-006**: 页面加载后1秒内完成布局渲染，三栏结构清晰可辨，各区域有明确标识。
- **SC-007**: 用户进入系统后默认显示当天日视图，无需手动切换即可看到当天所有时间盒。
- **SC-008**: 日/周/月视图切换后，数据在1秒内完成渲染，无明显闪烁或布局跳动。

## Assumptions

- USOM 类型、数据库 Schema、Repository 层已在 001-align-foundation 中对齐完成，可直接使用。
- MVP 阶段使用硬编码的测试用户 ID，不涉及认证系统。
- AI 解析使用已配置的 LLM 提供商（DashScope/DeepSeek/OpenAI/智谱），默认 DashScope。
- 时间盒只支持 create_timebox 动作，不支持编辑、删除、关联任务/习惯等高级功能。
- 记忆框架（Memory Framework）暂不实现，DerivedSignals 使用默认空值。
- 时间触发（planned→running、running→ended）暂不实现自动化，仅实现意图驱动的创建路径。
- 能量匹配规则暂不实现，仅在 Domain.onValidate 中做结构性验证。
- 优化需求（界面调整 + 追踪日志）的详细功能定义见 plan.md §Summary 和 tasks.md §US6-US9，plan.md 在本 feature 范围内作为 spec.md 的实现层补充。
- 日期计算基于用户浏览器本地时区，不做服务端时区转换。
