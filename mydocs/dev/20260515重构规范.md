## 二、意图驱动实施框架

要求：

- 已开发的几个 Domain，包括 timebox，habits，okrs，tasks，都需要按照这个优化后的 Intent 进行对接。



### 1. Intent Engine 的路由机制

Intent Engine 在路由阶段的工作是一个**有界分类任务**，不是开放式生成：

```
输入：用户原始 input
上下文：所有已注册 Domain 的 intent_triggers 列表

AI 任务：
  从 N 个 Domain × M 个 action 的候选集中，
  选出最匹配的 (targetDomain, action)，
  输出置信度 0-1

输出：
  { targetDomain: 'habits', action: 'create_habit', confidence: 0.92 }
  或
  { ambiguous: true, candidates: [...] }  // 置信度低时
```

这与 LLM function calling / tool selection 是同一个模式——你给模型描述工具，模型选一个，选完之后执行是确定性的。

关键约束是：**输出空间有界**（只能选已注册的 Domain 和 action），不是让 AI 自由生成执行计划。



### 2. Intent Engine 处理过程

Intent Engine 的处理分两个性质完全不同的阶段：

```
用户输入
  ↓
[阶段 A] 路由判断：这个意图属于哪个 Domain？哪个 action？
          → 本质是分类问题，AI 必须参与，天然是解释型
  ↓
[阶段 B] 字段补全：StructuredIntent 的 fields 是否完整？
          → 本质是结构化提取，AI 辅助，输出类型确定
  ↓
StructuredIntent（targetDomain + action + fields 全部确定）
  ↓
后续合约型链路（Rule Engine → State Machine ...）
```

阶段 A属于“解释型执行”，需要做判断，阶段B以后属于“合约型执行”，是必须遵循的流程。





### 3. 三类边界情况的处理

**情况 1：置信度低，路由不确定**

```
用户："帮我把每周健身计划整理一下"

候选：
  habits.create_habit     confidence: 0.61
  tasks.create_task       confidence: 0.55
  review.create_review    confidence: 0.30

处理：Intent Engine 向用户澄清
  → "你是想建立一个长期健身习惯，还是整理成一个一次性任务清单？"
```

**情况 2：跨 Domain 意图**

```
用户："我想每天早上跑步，作为健康目标的一部分"

识别出：
  habits.create_habit   （跑步）
  okrs.link_to_objective  （关联健康目标）

处理：拆解为两个 StructuredIntent，Orchestrator 顺序处理
```

这个拆解逻辑归 Intent Engine，不归任何单个 Domain。

**情况 3：用户使用斜杠快捷操作**

```
/habits:create_habit 每天跑步

处理：targetDomain='habits', action='create_habit' 直接确定，
      跳过 AI 路由分类，直接进入字段补全阶段
      → fastPath 路由，confidence 固定 1.0
```



### 4. Intent 的处理说明

```
用户: 输入Intent
  - 选择如下三种意图输入方式，自然语言、快捷方式（功能菜单的快捷方式，例如："/habits: create_habit"）、功能菜单（兜底选择），其中快捷方式（未加任何其他指令）、功能菜单可直接出发 template_form 表单录入模式

APP: 解析意图
  - 路由分类：找到匹配的 targetDomain.action，不清晰的 routingConfidence 低于 0.7(可用规则引擎配置)
  - 根据 Domain 的 manifest.yaml，提取关键信息，形成 StructuredIntent
  
APP: 展现信息
  - 如果用户选择 template_form 表单输入信息，直接跳过该过程
  - 获取action相关合适的Markdown模板文件，将 StructuredIntent 按照模板的形式展示
  - 提示用户需要补全的信息
  
用户: 确认意图
  - 可以通过编辑信息(Markdown形式)来补全
  - 确认信息（意图确认）
```



### 5. 基于 template_form 和 AI Markdown 的工作流优化

#### **必要性**

对于复杂的创建/规划操作（如创建一组 OKR + 关联 KR + 关联任务），通过读取附件生成Markdown 文档或者用户直接编辑 Markdown文档比逐步表单更适合用户协同编辑和 AI 辅助。这是对 `template_form` 路径的文档化扩展。

#### **Domain manifest 的定义**

- 所有 template_form 字段均在 Domain 中明确定义（可以是脚本）
- 所有 markdown 模板均在 Domain 中定义（文件形式）

#### **Markdown处理流程**

```
AI 生成 Markdown（包括解析上传的附件生成的Markdown）
  ↓ 基于 Domain manifest 的 markdownTemplate 定义
  
用户编辑 Markdown（协同修改，可能多轮）
  ↓ 这个 Markdown 是一个临时状态对象，需要有生命周期管理
  
用户确认 Markdown
  ↓
Intent Engine 解析 Markdown → 提取 StructuredIntent
  ↓ （可能是多个 StructuredIntent，批量处理）
后续走标准 Nexus 链路
```

#### **注意**

如果一个 Markdown 文件产生多个相互依赖的 StructuredIntent（如先创建 Objective，再创建引用该 Objective ID 的 KeyResult），Orchestrator 需要处理顺序依赖。

这个批处理逻辑在当前设计中没有定义，MVP 建议先限制 Markdown 工作流只产生单一 Domain 的操作，跨 Domain 批处理留后续版本。
