# APP核心框架梳理



## 本文目的

本文是在MVP开发中关于架构的重要补充文档，在基本维持原架构的同时，对部分架构进行修正和补充。

本文档作为架构调整的讨论，不作为最终开发依据。



## 本文内容

- 进一步明确 AI 意图驱动的四大机制

- 优化 Domain-plugin 的自描述能力和内涵

- state Machine 的设计缺陷改进

  



## 一、AI意图驱动的几大机制

主要包括如下几类形式：

### 1. 具体行动信息输入/修改

直接用自然语言直接输入明确的指令，例如增加、修改、删除一个时间盒计划、任务，或者打卡执行。也可以通过快捷指令、功能菜单启动具体功能，例如启动每日时间盒计划、日总结、周总结等。



### 2. 综合信息查询和统计

输入自然语言查询用户数据，包括筛选详细信息和统计信息，可生成数据报告模板和数据查询表单



### 3. 教练式指导

针对某个主题、数据或磁贴信息进行深度交流，提供教练式指导



几种形式可以相互转换，例如执行综合查询后，针对数据进行教练式对话。







##  三、State Machine 架构设计缺陷的改进



### 1. 架构问题

- 耦合性问题

  - 当前设计让 State Machine 成为全系统所有对象生命周期规则的执行者，这意味着：每新增一个 Domain，State Machine 就必须被修改——Task、Habit、Timebox、OKR 各有各的状态流转规则全写在 Nexus 里。这违反了 Domain Plugin 的可扩展性承诺，是耦合泄漏，不是"正常的架构复杂度"。

- 业务逻辑判断问题：

  - 当前的错误：

    State Machine（Nexus 组件）当前隐含地需要硬编码：

  ```
  if (objectType === 'Task') {
    // Task 的合法跃迁：draft→active, active→scheduled...
  } else if (objectType === 'Habit') {
    // Habit 的合法跃迁：draft→active, active→suspended...
  }
  ```

  这不是"State Machine 作为执行器"，这是"State Machine 作为业务知识仓库"。

  

  

### 2. 调整方案：生命周期声明下沉到 Domain Manifest

State Machine 应当是**通用生命周期执行器**，它不知道 Habit 有什么状态，它只知道"给我一个合法的跃迁声明，我来执行"。



- **在 Domain 的 manifest 增加如下声明**

  - 对象生命周期定义（一个Domain 可以有多个对象）

  - 对象生命周期转化规则定义，可考虑使用 from, to, trigger('intent'|'time') 来定义，intent = 经过 Orchestrator; time = State Machine 自行捕捉
  - 定义不可回退状态

- **State Machine 执行逻辑变更**：

  ```
  收到 StateProposal
    → 查找 targetObject 对应 Domain 的 manifest.lifecycle
    → 校验 from → to 跃迁是否在声明范围内
    → 校验 不可回退 约束
    → 合法则执行，拒绝非法跃迁
  ```
