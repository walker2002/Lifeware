# APP核心框架梳理



## 本文目的

本文是在MVP开发中关于架构的重要补充文档，在基本维持原架构的同时，对部分架构进行修正和补充。

本文档作为架构调整的讨论，不作为最终开发依据。



## 本文内容

- 进一步明确 AI 意图驱动的四大机制

- 优化 Domain-plugin 的自描述能力和内涵

- state Machine 的设计缺陷改进

  



## 一、AI意图驱动的几大机制

主要包括如下几类形式：

### 1. 具体行动信息输入/修改

直接用自然语言直接输入明确的指令，例如增加、修改、删除一个时间盒计划、任务，或者打卡执行。也可以通过快捷指令、功能菜单启动具体功能，例如启动每日时间盒计划、日总结、周总结等。



### 2. 综合信息查询和统计

输入自然语言查询用户数据，包括筛选详细信息和统计信息，可生成数据报告模板和数据查询表单



### 3. 教练式指导

针对某个主题、数据或磁贴信息进行深度交流，提供教练式指导



几种形式可以相互转换，例如执行综合查询后，针对数据进行教练式对话。



## 二、意图驱动实施框架

### 1. Domain manifest.yaml 的规范

所有的意图，通过 Nexus 的编排，都会落实到具体的 Domain plugin 的相关能力，各领域的能力包括：

Domain manifest 应该声明结构化的路由触发描述，供 Intent Engine 做 AI 分类的上下文，而不是自由格式的说明书。

原总体设计未提到这个机制，需要补充。

manifest 增加 `intent_triggers` 字段，示例 Domain habit 的 manifest.yaml 定义如下：

```yaml
# habit/manifest.yaml

domain_id: habits
version: "1.0.0"

intent_triggers:
  - action: create_habit
    description: "用户希望建立、养成、坚持某个长期周期性行为"
    examples:
      - "我想每天早上跑步"
      - "帮我养成读书的习惯"
      - "每周三次冥想"
    keywords: ["习惯", "坚持", "每天", "养成", "打卡", "每周"]
    signals:
      - "包含频率描述（每天/每周/每月）"
      - "包含长期持续的意图"
    excludes:
      - "一次性任务（做完即止）"
      - "有明确截止日期的工作"

  - action: log_habit
    description: "用户报告已完成某个习惯"
    examples:
      - "跑步完成了"
      - "今天读书打卡"
    signals:
      - "回报性语气，描述已发生的行为"

  - action: suspend_habit
    description: "用户暂时停止某个习惯"
    examples:
      - "出差期间暂停跑步"
```







### 2. Intent Engine 的路由机制

Intent Engine 在路由阶段的工作是一个**有界分类任务**，不是开放式生成：

```
输入：用户原始 input
上下文：所有已注册 Domain 的 intent_triggers 列表

AI 任务：
  从 N 个 Domain × M 个 action 的候选集中，
  选出最匹配的 (targetDomain, action)，
  输出置信度 0-1

输出：
  { targetDomain: 'habits', action: 'create_habit', confidence: 0.92 }
  或
  { ambiguous: true, candidates: [...] }  // 置信度低时
```

这与 LLM function calling / tool selection 是同一个模式——你给模型描述工具，模型选一个，选完之后执行是确定性的。

关键约束是：**输出空间有界**（只能选已注册的 Domain 和 action），不是让 AI 自由生成执行计划。





### 3. Intent Engine 处理过程

Intent Engine 的处理分两个性质完全不同的阶段：

```
用户输入
  ↓
[阶段 A] 路由判断：这个意图属于哪个 Domain？哪个 action？
          → 本质是分类问题，AI 必须参与，天然是解释型
  ↓
[阶段 B] 字段补全：StructuredIntent 的 fields 是否完整？
          → 本质是结构化提取，AI 辅助，输出类型确定
  ↓
StructuredIntent（targetDomain + action + fields 全部确定）
  ↓
后续合约型链路（Rule Engine → State Machine ...）
```

阶段 A属于“解释型执行”，需要做判断，阶段B以后属于“合约型执行”，是必须遵循的流程。

这部分这与之前的架构有补充。



### 4. 三类边界情况的处理

**情况 1：置信度低，路由不确定**

```
用户："帮我把每周健身计划整理一下"

候选：
  habits.create_habit     confidence: 0.61
  tasks.create_task       confidence: 0.55
  review.create_review    confidence: 0.30

处理：Intent Engine 向用户澄清
  → "你是想建立一个长期健身习惯，还是整理成一个一次性任务清单？"
```

**情况 2：跨 Domain 意图**

```
用户："我想每天早上跑步，作为健康目标的一部分"

识别出：
  habits.create_habit   （跑步）
  okrs.link_to_objective  （关联健康目标）

处理：拆解为两个 StructuredIntent，Orchestrator 顺序处理
```

这个拆解逻辑归 Intent Engine，不归任何单个 Domain。

**情况 3：用户使用斜杠快捷操作**

```
/habits:create_habit 每天跑步

处理：targetDomain='habits', action='create_habit' 直接确定，
      跳过 AI 路由分类，直接进入字段补全阶段
      → fastPath 路由，confidence 固定 1.0
```



### 5. Intent 的处理说明

```
用户: 输入Intent
  - 选择如下三种意图输入方式，自然语言、快捷方式（功能菜单的快捷方式，例如："/habits: create_habit"）、功能菜单（兜底选择），其中快捷方式（未加任何其他指令）、功能菜单可直接出发 template_form 表单录入模式

APP: 解析意图
  - 路由分类：找到匹配的 targetDomain.action，不清晰的 routingConfidence 低于 0.7(可用规则引擎配置)
  - 根据 Domain 的 manifest.yaml，提取关键信息，形成 StructuredIntent
  
APP: 展现信息
  - 如果用户选择 template_form 表单输入信息，直接跳过该过程
  - 获取action相关合适的Markdown模板文件，将 StructuredIntent 按照模板的形式展示
  - 提示用户需要补全的信息
  
用户: 确认意图
  - 可以选择编辑信息(AI Markdown形式)来补全
  - 确认信息（意图确认）
```



### 6. 基于 template_form 和 AI Markdown 的工作流优化

#### **必要性**

对于复杂的创建/规划操作（如创建一组 OKR + 关联 KR + 关联任务），通过读取附件生成Markdown 文档或者用户直接编辑 Markdown文档比逐步表单更适合用户协同编辑和 AI 辅助。这是对 `template_form` 路径的文档化扩展。

#### **Domain manifest 的定义**

- 所有 template_form 字段均在 Domain 中明确定义（可以是脚本）
- 所有 markdown 模板均在 Domain 中定义（文件形式）

#### **Markdown处理流程**

```
AI 生成 Markdown（包括解析上传的附件生成的Markdown）
  ↓ 基于 Domain manifest 的 markdownTemplate 定义
  
用户编辑 Markdown（协同修改，可能多轮）
  ↓ 这个 Markdown 是一个临时状态对象，需要有生命周期管理
  
用户确认 Markdown
  ↓
Intent Engine 解析 Markdown → 提取 StructuredIntent
  ↓ （可能是多个 StructuredIntent，批量处理）
后续走标准 Nexus 链路
```

#### **注意**

如果一个 Markdown 文件产生多个相互依赖的 StructuredIntent（如先创建 Objective，再创建引用该 Objective ID 的 KeyResult），Orchestrator 需要处理顺序依赖。

这个批处理逻辑在当前设计中没有定义，MVP 建议先限制 Markdown 工作流只产生单一 Domain 的操作，跨 Domain 批处理留后续版本。







##  三、State Machine 架构设计缺陷的改进



### 1. 架构问题

- 耦合性问题

  - 当前设计让 State Machine 成为全系统所有对象生命周期规则的执行者，这意味着：每新增一个 Domain，State Machine 就必须被修改——Task、Habit、Timebox、OKR 各有各的状态流转规则全写在 Nexus 里。这违反了 Domain Plugin 的可扩展性承诺，是耦合泄漏，不是"正常的架构复杂度"。

- 业务逻辑判断问题：

  - 当前的错误：

    State Machine（Nexus 组件）当前隐含地需要硬编码：

  ```
  if (objectType === 'Task') {
    // Task 的合法跃迁：draft→active, active→scheduled...
  } else if (objectType === 'Habit') {
    // Habit 的合法跃迁：draft→active, active→suspended...
  }
  ```

  这不是"State Machine 作为执行器"，这是"State Machine 作为业务知识仓库"。

  

  

### 2. 调整方案：生命周期声明下沉到 Domain Manifest

State Machine 应当是**通用生命周期执行器**，它不知道 Habit 有什么状态，它只知道"给我一个合法的跃迁声明，我来执行"。



- **在 Domain 的 manifest 增加如下声明**

  - 对象生命周期定义（一个Domain 可以有多个对象）

  - 对象生命周期转化规则定义，可考虑使用 from, to, trigger('intent'|'time') 来定义，intent = 经过 Orchestrator; time = State Machine 自行捕捉
  - 定义不可回退状态

- **State Machine 执行逻辑变更**：

  ```
  收到 StateProposal
    → 查找 targetObject 对应 Domain 的 manifest.lifecycle
    → 校验 from → to 跃迁是否在声明范围内
    → 校验 不可回退 约束
    → 合法则执行，拒绝非法跃迁
  ```
