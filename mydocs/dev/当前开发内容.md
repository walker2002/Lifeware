# 主题：重新检查Nexus相关组件违宪问题



### [000] Nexus.Orchestrator 问题

### 功能

Nexus.Orchestrator

### 状态

- 完成时间：

### 现状

在 executeIntent 方法中存在多处硬编码的 domainId 判断：

违反的宪章条款

1. Manifest Runtime Consumption (宪章 p.515-574)

核心原则 ： manifest.yaml 是运行时配置，Nexus 组件必须从注册表动态读取，不得硬编码。

相关规定 ：
 "Nexus components MUST read manifest values from the registry at runtime."
"Hardcoding detection — code smells indicating violation: Inline string literals matching manifest values used for comparison instead of reading from the loaded manifest"
当前问题 ：代码直接使用字符串字面量（如 'timebox' 、 'habits' ）进行比较，而非从 manifest 注册表读取。

2. Orchestrator Purity (宪章 p.403-436)

核心原则 ：Orchestrator 是纯调度器，不得包含业务逻辑。

相关规定 ：
 "The Orchestrator is a pure dispatcher. It MUST NOT contain business logic, write state, or call AI Runtime directly."
当前问题 ：硬编码的 domainId 判断本质上是业务路由逻辑，违反了 Orchestrator 的纯调度器角色定位。

3. Domain Plugin Dual-Track Model (宪章 p.152-294)

核心原则 ：添加新 Domain 不应修改 Nexus 组件。

相关规定 ：
 "Adding a new Domain MUST NOT require modifying Intent Engine routing logic or State Machine transition rules."
当前问题 ：每新增一个 Domain，都需要修改 Orchestrator 的 executeIntent 方法，违反了 Domain Plugin 的独立性承诺
